Question 1 (20 marks)

(a) (i) Outline two desirable qualities of a good program code design. (2 marks)

(ii) Outline three reasons that could influence a system designer to use functional decomposition. (3 marks)

(b) (i) State two skills expected of a systems analyst. (2 marks)

(ii) Differentiate between black-box and grey-box testing methods. (4 marks)

(c) Clifford intends to use system flow chart to analyze a proposed information system. Outline three symbols that he is likely to use in the chart. (3 marks)

(d) Irene was required to carry out training on a new system without interfering with the normal operations on the organization.

(i) Explain the most appropriate training method that she could use. (2 marks)

(ii) Explain two challenges that she could experience while using the training method identified in (i). (4 marks)

Answer

(a)(i) Qualities of Good Program Code Design:

  • Readability: Code is easy to understand and follow.
  • Maintainability: Code is easy to modify and update.
  • Efficiency: Code performs well and uses resources effectively.
  • Modularity: Code is broken down into logical, reusable components.

(a)(ii) Reasons for Using Functional Decomposition:

  • Manage Complexity: Breaks down large problems into smaller, manageable parts.
  • Improve Understanding: Clarifies the different functions and their relationships.
  • Facilitate Allocation: Allows different functions to be assigned to different developers/teams.
  • Promote Reusability: Identifies common functions that can be reused.

(b)(i) Skills Expected of a Systems Analyst:

  • Analytical Skills: Ability to understand problems and devise solutions.
  • Communication Skills: Ability to interact effectively with users, management, and technical staff.
  • Technical Skills: Understanding of IT concepts, databases, programming, etc.
  • Business Knowledge: Understanding of the organization's domain and processes.

(b)(ii) Black-Box vs Grey-Box Testing:

Black-Box Testing Grey-Box Testing
Tests functionality without knowledge of internal code structure. Tests with partial knowledge of internal structure (e.g., knowing database schema or algorithms used).
Focuses on inputs and outputs (what the system does). Focuses on specific internal paths or data structures based on limited knowledge.

(c) System Flowchart Symbols:

  • Process (Rectangle): Represents an operation or task.
  • Input/Output (Parallelogram): Represents data input or output.
  • Decision (Diamond): Represents a point where a choice is made.
  • Document (Rectangle with wavy bottom): Represents a printed document.
  • Flowline (Arrow): Shows the direction of flow.

(d)(i) Appropriate Training Method:

  • Phased Training or On-the-Job Training (OJT). Justification: These methods allow training to be integrated into the work schedule or conducted in stages, minimizing disruption to daily operations.

(d)(ii) Challenges of Phased/OJT Training:

  • Scheduling Conflicts: Difficult to pull staff away from duties for training sessions.
  • Inconsistent Training Quality: Depends on the trainer or mentor available during the work shift.
  • Distractions: Trainees may be interrupted by regular work tasks.

Question 2 (20 marks)

(a) Differentiate between investigation and fact recording as used in feasibility studies. (4 marks)

(b) Use case diagrams are popular in system analysis. Describe three advantages of the tool that could be influencing this trend. (6 marks)

(c) Paul intends to use data dictionaries to analyze a proposed information system. Outline four components of this tool. (4 marks)

(d) A lecturer described roles of system documents to a SAD class. Describe three roles that he could have mentioned. (6 marks)

Answer

(a) Investigation vs Fact Recording:

Investigation Fact Recording
Active process of gathering information (interviews, observation, etc.). Process of documenting the gathered information systematically.
Focuses on understanding the problem/system. Focuses on creating a reliable record of findings.

(b) Advantages of Use Case Diagrams:

  • Captures Functional Requirements: Clearly shows interactions between actors and the system for specific goals.
  • Easy Communication: Understandable by both technical and non-technical stakeholders.
  • Defines System Scope: Helps establish system boundaries by showing actor interactions.
  • Basis for Design & Testing: Guides design decisions and helps derive test cases.

(c) Components of a Data Dictionary:

  • Data Element Name: Unique identifier for the data item.
  • Description/Definition: Meaning of the data element.
  • Data Type & Length: E.g., Text(50), Integer, Date.
  • Aliases: Other names used for the data element.
  • Validation Rules: Constraints or rules the data must follow (e.g., range, format).
  • Source/Origin: Where the data comes from.

(d) Roles of System Documents:

  • Communication: Facilitate understanding between stakeholders (users, developers, managers).
  • Maintenance Guide: Provide information needed for future modifications and troubleshooting.
  • Training Material: Serve as a basis for training users and support staff.
  • Project Management: Help track progress, define scope, and manage changes.
  • System Audit: Provide a record for auditing and compliance purposes.

Question 3 (20 marks)

(a) Define the term Net Present Value (NPV) as used in project evaluation. (2 marks)

(b) Explain two factors that should be considered when creating terms-of-reference during problem identification. (4 marks)

(c) (i) Petro would like to design a deterministic system for the final year project. Explain two typical characteristics that could be exhibited by this type of a system. (4 marks)

(ii) Differentiate between risk and uncertainty as used in project management. (4 marks)

(d) Tom intends to test a system with modules that were developed and tested separately.

(i) Explain the most appropriate testing method that he could use to achieve his goal. (2 marks)

(ii) Explain two methods of carrying out the testing method identified in (i). (4 marks)

Answer

(a) Net Present Value (NPV):

  • A project evaluation technique that calculates the present value of future cash inflows minus the present value of cash outflows, using a specified discount rate. It determines the profitability of an investment.

(b) Factors for Creating Terms of Reference (ToR):

  • Problem Definition: Clearly stating the problem or opportunity the project aims to address.
  • Project Objectives: Defining specific, measurable goals for the project.
  • Scope & Boundaries: Outlining what is included and excluded from the project.
  • Constraints: Identifying limitations like budget, time, resources, or regulations.
  • Stakeholders: Identifying key individuals or groups involved or affected.

(c)(i) Characteristics of Deterministic System:

  • Predictable Output: The outcome is solely determined by the initial state and inputs.
  • No Randomness: The system's behavior does not involve chance or probability.

(c)(ii) Risk vs Uncertainty:

Risk Uncertainty
Potential future event with a known or estimable probability and impact. Potential future event where the probability or impact cannot be reasonably estimated.
Can be managed through mitigation, avoidance, transfer, acceptance. Harder to manage proactively; often requires contingency planning or adaptive strategies.

(d)(i) Appropriate Testing Method:

  • Integration Testing. Justification: Since modules were developed and unit tested separately, the next step is to test how these modules interact when combined.

(d)(ii) Methods for Integration Testing:

  • Big Bang Integration: All modules are combined at once and tested together. Simpler but makes error isolation difficult.
  • Top-Down Integration: Starts testing with top-level modules, using stubs for lower-level ones, then gradually integrates downwards.
  • Bottom-Up Integration: Starts testing with lowest-level modules, using drivers, then gradually integrates upwards.
  • Sandwich Integration: Combines top-down and bottom-up approaches.

Question 4 (20 marks)

(a) (i) Describe an expert system as used in information systems. (2 marks)

(ii) State two qualities of a good input form. (2 marks)

(b) (i) Outline two roles of a super user in an information system. (2 marks)

(ii) Differentiate between face-to-face interview and telephone interview. (4 marks)

(c) Dorothy intends to use an open loop control system in a proposed information system. Outline four attributes of this loop that could be exhibited in the system. (4 marks)

(d) An organization intends to procure a Management Information System. Explain three roles of this type of information system. (6 marks)

Answer

(a)(i) Expert System:

  • An information system that captures human expertise in a specific, limited domain to provide advice or solve problems, typically using a knowledge base and an inference engine.

(a)(ii) Qualities of a Good Input Form:

  • Clarity: Clear labels and instructions.
  • Logical Flow: Fields arranged in a sensible order.
  • Efficiency: Minimizes required input effort.
  • Error Prevention: Uses controls like dropdowns, validation.
  • Consistency: Follows standard layout and terminology.

(b)(i) Roles of a Super User:

  • Provides first-level support to other users within their department.
  • Acts as a liaison between end-users and the IT department/support team.
  • Often involved in testing new features or system updates.
  • May assist with user training.

(b)(ii) Face-to-Face vs Telephone Interview:

Face-to-Face Interview Telephone Interview
Allows observation of non-verbal cues. Lacks visual cues.
Easier to build rapport. Rapport building can be more challenging.
Better for complex or sensitive topics. Less suitable for complex/sensitive topics.
Can be more time-consuming and costly (travel). More cost-effective and quicker to arrange.

(c) Attributes of Open Loop Control System:

  • No Feedback: Output is not measured or used to adjust the input/process.
  • Simple & Inexpensive: Generally simpler design and lower cost.
  • Relies on Calibration: Accuracy depends on initial setup and assumes predictable behavior.
  • Susceptible to Disturbances: Cannot automatically correct for unexpected changes or errors.

(d) Roles of Management Information System (MIS):

  • Reporting: Provides routine summary and exception reports based on operational data.
  • Monitoring & Control: Helps managers track performance against goals and identify deviations.
  • Tactical Decision Support: Provides information to support structured and semi-structured decisions at the middle management level.

Question 5 (20 marks)

(a) Outline two signs of system obsolescence. (2 marks)

(b) Explain two reasons for replenishing resources in a project. (4 marks)

(c) True-Life Company Ltd. intends to install new information to replace their entire legacy system as a matter of urgency.

(i) Identify the most appropriate changeover strategy that you could use justifying your answer. (2 marks)

(ii) Explain two possible risks that the company would realize while using the changeover strategy identified in (i). (4 marks)

(d) Read the following narrative to answer the question that follows. Jasho Sacco opens new branches in major towns by registering the details (branch code and branch name), which are stored in a branch file at the head office. The branches register new members by processing applications from prospective members whose details are then stored in the members file. The customer service officer is then required to open a FOSA account of the new Sacco membersâ€™ and issue a Sacco membership card. When a Sacco member wants to make a transaction at the branch, he/she is required to produce a Membership card and national identity card for verification purposes. The member is then required to specify the type of transaction to be carried out. On completion the member is issued a transaction receipt and the transaction file updated.

Draw a level 1 dataflow diagram to model the narrative into a system. (8 marks)

Answer

(a) Signs of System Obsolescence:

  • High Maintenance Costs: Frequent repairs and difficulty finding parts/support.
  • Poor Performance: System is slow or cannot handle current workload.
  • Lack of Vendor Support: Original vendor no longer supports the hardware/software.
  • Incompatibility: Cannot integrate with newer technologies or systems.
  • Doesn't Meet Business Needs: Lacks features required by current business processes.

(b) Reasons for Replenishing Project Resources:

  • Consumption: Resources (materials, budget) are used up during project execution.
  • Scope Changes: Additional scope may require more resources than initially planned.
  • Addressing Risks/Issues: Unforeseen problems may consume contingency resources or require additional funding/personnel.
  • Inaccurate Initial Estimation: Original resource planning was insufficient.

(c)(i) Appropriate Changeover Strategy:

  • Direct Changeover (Cutover). Justification: The requirement is to replace the *entire* legacy system urgently. This strategy involves stopping the old system and starting the new one immediately, which is the fastest approach, albeit risky.

(c)(ii) Risks of Direct Changeover:

  • System Failure: If the new system fails upon launch, there is no fallback, causing major disruption.
  • Data Issues: Errors during data conversion/migration might only become apparent after going live.
  • User unpreparedness: Insufficient time for training or adaptation can lead to errors and reduced productivity.
  • High Stress: Puts immense pressure on the implementation team and users.

Question 6 (20 marks)

(a) Outline three disadvantages of using waterfall model in systems development. (3 marks)

(b) (i) Describe Structured English as used in system design. (2 marks)

(ii) Outline three signs of derailing projects. (3 marks)

(c) Jambo Football Academy intends to select a team for a tournament. A team member should be at least 4.3 feet tall and weigh at least 50 kilograms with a clean bill of health. Draw a limited entry decision table to represent the team selection processes. (6 marks)

(d) Julie was required to implement various keys in proposed database systems. Explain three types of keys that she could have selected. (6 marks)

Answer

(a) Disadvantages of Waterfall Model:

  • Inflexibility: Difficult and costly to go back to previous phases once completed.
  • Late Feedback: Users see the working system only late in the process, making requirement changes hard.
  • Risk: Assumes requirements are fully understood and stable upfront, which is often unrealistic.
  • Delayed Delivery: Working software is not produced until late in the cycle.

(b)(i) Structured English:

  • A tool used in structured analysis and design to describe process logic using a limited vocabulary of action verbs, nouns from the data dictionary, and basic control structures (Sequence, Selection (IF-THEN-ELSE), Iteration (WHILE, FOR)). It resembles pseudocode but is more readable for non-programmers.

(b)(ii) Signs of Derailing Projects:

  • Missed Milestones/Deadlines: Consistently failing to meet scheduled targets.
  • Budget Overruns: Spending more money than allocated.
  • Scope Creep: Uncontrolled expansion of project requirements.
  • Low Team Morale/High Turnover: Indicates underlying problems.
  • Poor Communication: Lack of clarity or regular updates among stakeholders.

(d) Types of Database Keys:

  • Primary Key: Uniquely identifies each record in a table (e.g., StudentID in a Students table). Ensures no duplicate records.
  • Foreign Key: A field in one table that links to the Primary Key of another table, enforcing relationships (e.g., CourseID in an Enrollments table linking to the Courses table).
  • Candidate Key: Any attribute or set of attributes that could potentially serve as a Primary Key (uniquely identifies records).
  • Alternate Key: A Candidate Key that was not chosen to be the Primary Key.
  • Composite Key: A Primary Key made up of two or more attributes combined.
  • Super Key: Any set of attributes that uniquely identifies a record (includes Primary Key plus potentially other attributes).

(a) Outline two applications of cloud computing in systems analysis and design. (2 marks)

(b) With the aid of a symbol, outline each of the following type of nodes in a decision tree:

(i) chance/event; (2 marks)

(ii) terminal/edge; (2 marks)

(iii) decision. (2 marks)

(d) (i) Break through analysis is normally used as a project evaluation tool. Explain two of its advantages. (4 marks)

(ii) An information system project has stakeholders who should do a follow up for its success. Outline four examples of such stakeholders. (4 marks)

Answer

(a) Cloud Computing Applications in SAD:

  • Collaboration Tools: Using cloud-based platforms (like Google Workspace, Microsoft 365) for document sharing, communication, and project management during analysis and design.
  • Prototyping/Development Environments: Utilizing cloud infrastructure (IaaS/PaaS) to quickly set up and scale development and testing environments without managing physical hardware.
  • Cloud-Based CASE Tools: Using modeling and design tools hosted in the cloud.

(c) Types of Project Risks:

  • Technical Risks: Issues related to technology feasibility, integration problems, performance shortfalls, security vulnerabilities.
  • Schedule Risks: Delays caused by inaccurate time estimates, resource unavailability, scope creep, unforeseen dependencies.
  • Budget Risks: Cost overruns due to poor estimation, scope changes, unexpected expenses.
  • Operational Risks: Problems with user acceptance, inadequate training, disruption to business processes after deployment.
  • Resource Risks: Lack of skilled personnel, insufficient tools or infrastructure.

(d)(i) Advantages of Break-Even Analysis:

  • Simplicity: Easy to understand and calculate the point where costs equal revenue.
  • Decision Point: Helps determine the minimum level of sales/activity needed to avoid losses.
  • Sensitivity Analysis: Can be used to see how changes in costs or prices affect the break-even point.

(d)(ii) Examples of Stakeholders for Follow-Up:

  • Project Sponsor/System Owner: Tracks benefits realization and ROI.
  • End Users: Provide feedback on usability and ongoing issues.
  • Operational Managers: Monitor impact on departmental performance and processes.
  • IT Support/Maintenance Team: Track system stability, performance, and maintenance effort.
  • Internal/External Auditors: Ensure compliance and control objectives are met.

Question 8 (20 marks)

(a) (i) Explain two types of systems documents that could be found in system development life cycle. (4 marks)

(ii) Explain the term perpetual license as applied in software acquisition. (2 marks)

(b) S&S Company Ltd. intends to redesign the input form for its information system to allow users capture data easily.

(i) Assume that you are hired as a consultant; describe the type of maintenance that you would recommend. (2 marks)

(ii) Explain two precautions that should be considered while using the maintenance method identified in (i). (4 marks)

Answer

(a)(i) Types of System Documents:

  • Requirements Specification: Details what the system must do (functional and non-functional requirements). Created during analysis.
  • Design Specification: Describes how the system will be built (architecture, database design, UI design, module specifications). Created during design.
  • Test Plan/Report: Outlines testing strategy, test cases, and documents test results. Created during testing.
  • User Manual: Guides end-users on how to operate the system. Created during implementation/deployment.
  • Technical Manual: Provides technical details for installation, configuration, and maintenance. Created during implementation/deployment.

(a)(ii) Perpetual License:

  • A software license that grants the user the right to use a specific version of the software indefinitely, usually after a one-time payment. It may or may not include rights to future updates or support.

(b)(i) Recommended Maintenance Type:

  • Perfective Maintenance. Justification: Redesigning an input form to improve usability ("capture data easily") is an enhancement to improve system performance or user experience, which falls under perfective maintenance.

(b)(ii) Precautions for Perfective Maintenance:

  • Impact Analysis: Carefully assess how changes to the input form might affect other parts of the system or existing data.
  • Thorough Testing: Test the redesigned form extensively, including regression testing, to ensure it works correctly and hasn't introduced new issues.
  • User Training/Communication: Inform users about the changes and provide necessary guidance or training on the new form layout/functionality.
  • Version Control: Keep track of changes made so they can be rolled back if necessary.
