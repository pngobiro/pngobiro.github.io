Question 1

(a) Explain the following terms as applied in system theory:

(i) Entropy;

(ii) Feedback. (4 marks)

(b) Outline four objectives of system analysis. (4 marks)

(c) Differentiate between cybernetic and deterministic systems stating an example in each case. (4 marks)

(d) With the aid of a flowchart, describe the process of developing a system using the prototype. (8 marks)

Answer

(a)

(i) Entropy: The tendency of a system to move towards disorder, decay, or disorganization over time if left without intervention or input of energy/information.

(ii) Feedback: Information about a system's output or performance that is returned to the system as input, allowing it to monitor its state and make adjustments to achieve its goals.

(b) Objectives of system analysis:

  1. To understand the existing system thoroughly (processes, data, problems).
  2. To identify and define the problems or opportunities that the new system should address.
  3. To determine and document the requirements (functional and non-functional) for the proposed system.
  4. To evaluate the feasibility (technical, economic, operational) of potential solutions.

(c) Cybernetic vs Deterministic Systems:

Cybernetic SystemsDeterministic Systems
Self-regulating systems that use feedback to adjust their behavior towards a goal.Systems where the output is fully determined by the input and initial state; behavior is predictable.
Example: A thermostat maintaining room temperature.Example: A simple mechanical clock or a basic calculation (2+2=4).

Question 2

(a) (i) State three components of a data dictionary citing an appropriate example for each. (3 marks)

(ii) Distinguish between feasibility study and system study. (4 marks)

(b) Explain the term soft system thinking. (2 marks)

(c) Figure 1 shows the relationship of modules in a particular system.

            Main
           /    \
      Module C   Module A
         |
      Module B

(i) Identify the span of control for module A. (2 marks)

(ii) Explain the term fan-in in relation to module B. (3 marks)

(d) Describe the application of the following types of coupling in a system stating one disadvantage in each case:

(i) Content;

(ii) Control. (6 marks)

Answer

(a) (i) Data Dictionary Components:

  1. Data Element: Definition of a basic data item. Example: `CustomerID` (Integer, Primary Key, Description: Unique identifier for a customer).
  2. Data Structure: A group of related data elements. Example: `CustomerAddress` (Street, City, PostalCode).
  3. Data Flow: Movement of data between processes, data stores, or external entities. Example: `PaymentDetails` (Flows from Customer to Process Payment).
  4. Data Store: A repository where data is held. Example: `CustomerFile` (Stores all customer records).
*(Note: Only three were required)*

(ii) Feasibility Study vs System Study:

Feasibility StudySystem Study (Analysis)
Preliminary investigation to determine if a proposed project is viable (technically, economically, operationally).Detailed investigation of an existing system to understand its processes, problems, and requirements for improvement.
Asks: "Should we do the project?"Asks: "What does the current system do, and what should the new system do?"
Outcome: Recommendation to proceed, modify, or abandon the project.Outcome: Detailed requirements specification for the new/modified system.

(b) Soft System Thinking (SST): An approach to organizational process modeling and problem-solving, particularly useful for complex, ill-defined problems involving human perspectives and social dynamics. It uses conceptual models (like Rich Pictures) to explore different viewpoints and facilitate discussion towards actionable improvements, rather than focusing solely on technical solutions.

(c) Module Relationships:

(i) Span of Control for Module A: Zero (0). Module A does not call or control any subordinate modules in the given diagram.

(ii) Fan-in for Module B: Fan-in refers to the number of modules that call or invoke a particular module. In this diagram, Module B is called only by Module C. Therefore, the fan-in for Module B is one (1).

(d) Types of Coupling:

(i) Content Coupling:

  • Application: One module directly modifies or relies on the internal data or implementation details of another module. (e.g., Module X directly changing a variable inside Module Y).
  • Disadvantage: Very high dependency; changes in one module almost always require changes in the other, making maintenance extremely difficult and violating encapsulation. (Worst type of coupling).

(ii) Control Coupling:

  • Application: One module passes control information (e.g., a flag or code) to another module, influencing its execution path. (e.g., Module X passes a 'PrintMode' flag to Module Y).
  • Disadvantage: The calling module needs to know about the internal logic of the called module, reducing independence and making modules less reusable in different contexts.

(ii) Explain two uses of documentation. (2 marks)

(b) Distinguish between operational level and tactical level information. (4 marks)

(c) Judith would like to collect the following information about a system she intends to develop:

  • live data which will be used to test the system;
  • the end users opinion about the introduction of the new system;
  • details of the operations in the organization.

For each of the above cases, identify the most appropriate fact finding technique she would apply justifying your answer. (6 marks)

*(Note: The question asks for Payback Method but provides a discount rate, suggesting NPV might be intended. Both calculations shown below for clarity)*

Answer

(a) (i) Qualities of Good System Documentation:

  1. Accurate & Complete: Reflects the system correctly and covers all necessary aspects.
  2. Clear & Understandable: Written in a way that the intended audience can easily comprehend.
  3. Well-Organized & Accessible: Structured logically with indexes/TOC for easy navigation.
  4. Up-to-Date & Maintainable: Easy to revise as the system changes.
*(Note: Only four were required)*

(ii) Uses of Documentation:

  1. System Maintenance: Provides reference for developers fixing bugs or adding features.
  2. User Training & Support: Helps users learn how to use the system and troubleshoot issues.
  3. Communication: Facilitates understanding between stakeholders (users, developers, managers).
*(Note: Only two were required)*

(b) Operational vs Tactical Level Information:

Operational Level InformationTactical Level Information
Detailed, transaction-oriented data about daily activities.Summarized information derived from operational data, used for short-to-medium term planning and control.
Focus: Efficiency of specific tasks.Focus: Performance monitoring, resource allocation, achieving departmental goals.
Timeframe: Current, real-time.Timeframe: Weekly, monthly, quarterly summaries/trends.
Example: Individual sales records, production line status.Example: Monthly sales reports by region, budget variance analysis.

(c) Fact Finding Techniques:

  1. Live data for testing:
    • Technique: Record Sampling / Existing Data Analysis.
    • Justification: Directly obtains actual data used in current operations, which is ideal for realistic system testing.
  2. End users' opinion:
    • Technique: Interviews or Questionnaires.
    • Justification: Directly gathers subjective views, concerns, and expectations from the people who will use the system. Interviews allow probing, while questionnaires reach more people.
  3. Details of operations:
    • Technique: Observation or Document Analysis.
    • Justification: Observation shows how tasks are actually performed in real-time, while document analysis reveals the formal procedures and data involved.

Question 5

(a) Explain the following terms as applied in SSM:

(i) Rich picture;

(ii) Root definition. (4 marks)

(b) (i) With the aid of a diagram, describe the phased approach to system development.

(ii) Explain two drawbacks of the phased approach. (10 marks)

(c) (i) Define the term Rapid Application Development. (2 marks)

(ii) Explain two principles of risk management during development of a new system. (4 marks)

Answer

(a) SSM Terms:

(i) Rich Picture: A freehand drawing or diagram used in Soft Systems Methodology (SSM) to capture the complexities of a problematic human situation. It includes structures, processes, relationships, conflicts, concerns, and subjective elements without imposing a formal structure, aiming to provide a holistic view.

(ii) Root Definition: A concise, structured statement defining the core purpose and nature of a relevant system identified within the problem situation in SSM. It typically follows the CATWOE mnemonic (Customers, Actors, Transformation, Weltanschauung/Worldview, Owners, Environmental constraints) to ensure key elements are considered.

(b) (i) Phased Approach (Phased Implementation/Conversion):

    +--------------+     +--------------+     +--------------+
    | Old System   | --> | Phase 1 New  | --> | Phase 2 New  | --> etc.
    |              |     | System Module|     | System Module|
    +--------------+     +------+-------+     +------+-------+
                                |                    |
                                ▼                    ▼
                           Evaluate             Evaluate
                           & Refine             & Refine

Description: A system changeover strategy where the new system is implemented incrementally in stages or modules (phases). Each phase replaces a part of the old system or introduces functionality to a specific user group. The old system continues to operate alongside the implemented phases until the entire new system is rolled out.

(ii) Drawbacks of Phased Approach:

  1. Extended Transition Period: The organization operates with both old and new systems (or parts) concurrently for a longer duration, which can be confusing for users and complex to manage.
  2. Integration Challenges: Ensuring seamless integration and data consistency between the remaining parts of the old system and the newly implemented phases can be technically difficult and require temporary interfaces (bridges).
  3. Delayed Full Benefits: The full benefits of the new system are not realized until all phases are completed, which might take a significant amount of time.
*(Note: Only two were required)*

(c) (i) Rapid Application Development (RAD): A software development methodology that emphasizes speed and user involvement through techniques like iterative development, prototyping, workshops (like JAD), and the use of CASE tools to quickly deliver functional systems.

(ii) Principles of Risk Management:

  1. Risk Identification & Assessment: Proactively identify potential threats or problems (risks) early in the project and analyze their likelihood and potential impact.
  2. Risk Mitigation/Control: Develop and implement strategies to reduce the likelihood or impact of identified risks (e.g., contingency plans, prototyping, using proven technology, staff training).
  3. Risk Monitoring: Continuously track identified risks and look for new ones throughout the project lifecycle, adjusting mitigation plans as needed.
*(Note: Only two were required)*

(a) (i) Outline four causes of system obsolescence. (4 marks)

(ii) Explain the term system enhancement. (2 marks)

(b) With the aid of diagrams, distinguish between phased and parallel change over techniques. (6 marks)

(c) The following are the activities that create, affect and remove an entity called student in a college management information system:

  • Before admission, a student is registered by being given an admission number then proceeds to pay the fees.
  • The student can then decide to reside in the college by paying boarding fees or opt for non-residence status.
  • During learning, a student concurrently attends a number of lectures and examinations.
  • At the end of the course, a student must be cleared by the institute.

(i) Draw ELH to exhaustively model the entity. (6 marks)

(ii) Include the state indicators on the diagram in c(i). (2 marks)

Answer

(a) (i) Causes of System Obsolescence:

  1. Technological Advancements: New hardware or software makes the current system outdated or incompatible.
  2. Changing Business Needs: The system no longer supports evolving business processes or requirements.
  3. High Maintenance Costs: The cost of fixing bugs and keeping the old system running becomes prohibitive.
  4. Poor Performance: The system becomes too slow or unreliable to meet user demands.
  5. Lack of Vendor Support: The original vendor no longer supports the hardware or software.
*(Note: Only four were required)*

(ii) System Enhancement: Modifying an existing system to add new functionalities, improve performance, or adapt it to new requirements, without completely replacing it. It's a form of perfective or adaptive maintenance.

(b) Phased vs. Parallel Changeover:

Phased Changeover:

    Time --->

    | Old System   |     | Old System   |

Description: New system modules are introduced incrementally, replacing parts of the old system over time.

Parallel Changeover:

    Time --->
    +--------------+==================> (Stop Old)
    | Old System   |
    +--------------+
    +--------------+==================> (Continue New)
    | New System   |
    +--------------+
          | Compare Outputs |

Description: Both old and new systems run concurrently for a period. Outputs are compared for verification before the old system is stopped.

Question 7

(a) Explain the following terms as applied in software systems:

(i) Robustness;

(ii) Maintainability. (4 marks)

(b) During evaluation of an information system, the following factors were considered about the interface:

  • Whether all the information needed by a user appears on the screen;
  • Time used to carry out a task, system recovery or the failure rate of the system;
  • Mental effort required by the user to use the interface;
  • Ease of learning to use the interface and help incorporated.

For each factor, identify the usability metric considered by the system evaluators. (4 marks)

(c) (i) Outline four factors that should be considered when designing processes in a system. (2 marks)

(ii) Distinguish between processing and organizational file requirements. (2 marks)

*(Note: Marks allocated seem low for the number of items requested)*

Answer

(a) Software Systems Terms:

(i) Robustness: The ability of a system to handle errors during execution or invalid input gracefully, without crashing or producing incorrect results. It measures resilience to unexpected conditions.

(ii) Maintainability: The ease with which a software system can be modified to correct faults, improve performance, adapt to a changed environment, or add new features.

(b) Interface Usability Metrics:

  1. Information needed appears on screen: **Effectiveness** (Can users achieve their goals accurately and completely?) or **Content Clarity**.
  2. Time used, recovery, failure rate: **Efficiency** (How quickly and with what resources can users achieve goals?) and **Reliability**.
  3. Mental effort required: **Satisfaction** (Subjective user experience) or **Cognitive Load**.
  4. Ease of learning and help: **Learnability**.

(c) (i) Factors in Process Design:

  1. Inputs & Outputs: What data is needed and what results are produced.
  2. Logic & Rules: The steps and conditions involved in the transformation.
  3. Performance Needs: Required speed, throughput, or response time.
  4. Error Handling: How exceptions or invalid data are managed.
  5. Security Controls: Access rights and data protection within the process.
*(Note: Only four were required, but marks suggest maybe only two were expected)*

(ii) Processing vs Organizational File Requirements:

Processing File RequirementsOrganizational File Requirements
Specify how data needs to be accessed, manipulated, and transformed *during* a process (e.g., sorting order, temporary storage needs).Specify how data needs to be structured, stored, and managed long-term within the organization's data architecture (e.g., file types, storage media, backup frequency, access permissions).

Question 8

(a) State four guidelines of system testing. (2 marks)

*(Note: Marks allocated seem low for four items)*

(b) Describe two types of test data citing a reason for including each in a test plan. (4 marks)

(c) With the aid of diagrams, describe object structure as applied in system modeling. (4 marks)

*(Note: This repeats Q6(c) from 2008nov - likely an error in the original paper, but answered as asked)*

Answer

(a) System Testing Guidelines:

  1. Plan testing early in the lifecycle.
  2. Testing should aim to find errors, not prove correctness.
  3. Use both valid and invalid test data.
  4. Test boundary conditions.
  5. Document test cases and results thoroughly.
  6. Independent testing is often more effective.
*(Note: Only four were required, marks seem low)*

(b) Types of Test Data:

  1. Valid (Normal) Data: Input that falls within the expected range and format.
    • Reason: To verify that the system processes correct inputs as expected according to requirements.
  2. Invalid (Erroneous) Data: Input that falls outside the expected range or format (e.g., negative numbers where positive expected, text in a numeric field).
    • Reason: To verify that the system handles errors gracefully, provides appropriate messages, and does not crash.
  3. Boundary Data: Input at the edges of valid ranges (e.g., minimum/maximum allowed values).
    • Reason: Errors often occur at boundaries, so testing these specifically increases confidence.
*(Note: Only two types with reasons were required)*

(c) Object Structure in System Modeling:

Object structure modeling focuses on the static aspects of a system, showing the classes, their attributes, operations, and the relationships between them. Key diagrams include:

1. Class Diagram: Shows the blueprint for objects.

    +-----------------+          +-----------------+
    |   ClassA        |          |    ClassB       |
    +-----------------+          +-----------------+
    | - attribute1    |───(Rel)──| - attributeX    |  (Rel = Relationship e.g., Association)
    | - attribute2    |          |                 |
    +-----------------+          +-----------------+
    | + operation1()  |          | + operationY()  |
    | + operation2()  |          |                 |
    +-----------------+          +-----------------+

2. Object Diagram: Shows instances of classes and their links at a specific point in time.

    +-----------------+          +-----------------+
    | objectA : ClassA|          | objectB : ClassB|
    +-----------------+          +-----------------+
    | attribute1 = val|----------| attributeX = val|  (Link = Instance of Relationship)
    | attribute2 = val|          |                 |
    +-----------------+          +-----------------+

Description: These diagrams illustrate the 'things' (classes/objects) in the system, what they know (attributes), what they can do (operations), and how they are connected (relationships/links).
