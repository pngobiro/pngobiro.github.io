Question 1 (20 marks)

(a) (i) Outline two components of an information system. (2 marks)

(ii) Outline two roles of a designer in an organization. (2 marks)

(b) Explain each of the following types of information systems:

(i) open; (1 mark)

(ii) closed; (1 mark)

(iii) probabilistic; (1 mark)

(iv) deterministic. (1 mark)

(c) (i) Differentiate between data and information as used in systems analysis. (4 marks)

(d) With the aid of an example in each case, describe the following type of questions that could be used when designing a questionnaire:

(i) open; (4 marks)

(ii) closed. (3 marks)

Answer

(a)(i) Components of Information System:

  • Hardware
  • Software
  • Data
  • People
  • Processes/Procedures
  • Network

(a)(ii) Roles of a Designer:

  • Translating user requirements into technical specifications.
  • Creating system architecture and component designs.
  • Ensuring design meets usability, performance, and security standards.

(b) Types of Information Systems:

  • (i) Open System: Interacts with its environment, receiving inputs and producing outputs (e.g., most business systems).
  • (ii) Closed System: Does not interact with its environment (theoretical, rare in practice).
  • (iii) Probabilistic System: Outcome cannot be precisely predicted even if inputs are known (e.g., weather forecasting system).
  • (iv) Deterministic System: Outcome can be precisely predicted if inputs and initial state are known (e.g., a simple calculation program).

(c)(i) Data vs Information:

  • Data: Raw, unprocessed facts and figures without context (e.g., 100, "Red", 2024-03-26).
  • Information: Processed, organized, and contextualized data that provides meaning and value (e.g., "Sales increased by 100 units", "The alert status is Red", "Order date is March 26, 2024").

(d) Questionnaire Question Types:

  • (i) Open Questions: Allow respondents to answer in their own words, providing detailed qualitative insights. Example: "What are your suggestions for improving the current system?"
  • (ii) Closed Questions: Provide predefined answer choices (e.g., Yes/No, multiple choice, rating scale), yielding quantitative, easily analyzable data. Example: "Do you find the system easy to use? (Yes/No)" or "Rate the system speed: (1-Poor, 5-Excellent)".

Question 2 (20 marks)

(a) (i) State two types interviews that could be used during fact finding. (1 mark)

(ii) Differentiate between operational and legal feasibility studies. (4 marks)

(b) Each of SDLC stage must produce an output. Outline the deliverable for each of the following stages:

(i) planning; (1 mark)

(ii) feasibility; (1 mark)

(iii) analysis; (1 mark)

(iv) design; (1 mark)

(v) implementation. (1 mark)

(c) Alex was required to write a report on top down analysis methodology. Outline four advantages that he is likely to have included in the report. (4 marks)

(d) Axor Company Ltd. intends to use structured method to analyze a proposed information system. As the systems analyst, explain three factors you could consider before using this method. (6 marks)

Answer

(a)(i) Types of Interviews:

  • Structured Interview
  • Unstructured Interview
  • Semi-structured Interview

(a)(ii) Operational vs Legal Feasibility:

Operational FeasibilityLegal Feasibility
Assesses if the proposed system will work effectively within the organization's existing environment, culture, and processes.Assesses if the proposed system complies with all relevant laws, regulations, and contractual obligations.
Considers user acceptance, training needs, organizational impact.Considers data privacy laws, licensing, labor laws, etc.

(b) SDLC Stage Deliverables:

  • (i) Planning: Project Plan / Project Proposal / Initial Feasibility Report.
  • (ii) Feasibility: Feasibility Study Report.
  • (iii) Analysis: System Requirements Specification / Requirements Definition Document.
  • (iv) Design: System Design Specification (including technical architecture, database design, UI design).
  • (v) Implementation: Tested System / Deployed System / User Manuals / Training Materials.

(c) Advantages of Top-Down Analysis:

  • Systematic Approach: Breaks down complex problems into smaller, manageable parts.
  • Clear Overview: Provides a high-level understanding of the system before diving into details.
  • Improved Control: Allows for better management and tracking of different system components.
  • Modular Design: Facilitates the development of independent modules.

(d) Factors for Using Structured Method:

  • Clarity of Requirements: Structured methods work best when requirements are relatively well-defined and stable.
  • Project Size and Complexity: Suitable for large, complex projects where a systematic breakdown is beneficial.
  • Team Experience: The team should be familiar with structured techniques (like DFDs, ERDs, Structured English).
  • Need for Rigorous Documentation: Structured methods emphasize formal documentation at each stage.

Question 4 (20 marks)

(a) (i) Explain two limitations of systems development life cycle. (4 marks)

(ii) Explain the term iteration as used in system development life cycle. (2 marks)

(b) Are-Soft Ltd. was hired to train employees of a certain organization on how to use a newly implemented information system such that the normal operations are not disrupted.

(i) Identify the most appropriate training method that could be used. (1 mark)

(ii) Outline four advantages of the training method identified in (i). (4 marks)

(c) Table 1 shows net profits for project A and B respectively, the initial cost for the two projects was Kshs. 180, 000 and Kshs. 220, 000 respectively. Use it to answer the questions that follow.

Table 1

(i) Determine the most worthwhile project using the return-on-investment technique. (5 marks)

(ii) Calculate the payback time for each project. (4 marks)

Answer

(a)(i) Limitations of SDLC (Traditional Waterfall):

  • Rigidity: Difficult and costly to go back to previous phases once a phase is completed.
  • Late Feedback: Users typically see the working system only late in the process, making requirement changes problematic.
  • Documentation Heavy: Can generate extensive documentation that may be time-consuming to produce and maintain.

(a)(ii) Iteration:

  • Repeating one or more phases of the SDLC in cycles. Each cycle produces a refined or expanded version of the system or a component, allowing for incremental development and feedback. Common in Spiral and Agile models.

(b)(i) Training Method:

  • Phased Training or Pilot Training. (Training a small group first, or training department by department).

(b)(ii) Advantages:

  • Minimal Disruption: Allows normal operations to continue while training occurs in stages or isolated groups.
  • Feedback Loop: Allows refinement of training materials and approach based on initial group's experience.
  • Reduced Risk: Issues with the system or training can be identified and addressed with a smaller group before wider rollout.
  • Creates Internal Experts: Pilot group members can assist colleagues later.

(c)(i) Return on Investment (ROI):

Project A: Total Profit = 20+30+50+65+70 = 235,000. Avg Profit = 235,000/5 = 47,000. ROI = (47,000 / 180,000) * 100% = 26.11%
Project B: Total Profit = 40+45+60+60+55 = 260,000. Avg Profit = 260,000/5 = 52,000. ROI = (52,000 / 220,000) * 100% = 23.64%
Project A is more worthwhile based on ROI.

(c)(ii) Payback Period:

Project A (Cost 180k):
Year 1: 20k (Cumulative: 20k)
Year 2: 30k (Cumulative: 50k)
Year 3: 50k (Cumulative: 100k)
Year 4: 65k (Cumulative: 165k)
Year 5: Need 180 - 165 = 15k. Payback = 4 + (15,000 / 70,000) = 4.21 years.

Project B (Cost 220k):
Year 1: 40k (Cumulative: 40k)
Year 2: 45k (Cumulative: 85k)
Year 3: 60k (Cumulative: 145k)
Year 4: 60k (Cumulative: 205k)
Year 5: Need 220 - 205 = 15k. Payback = 4 + (15,000 / 55,000) = 4.27 years.

Question 5 (20 marks)

(a) Outline two types of maintenance that could be applied on an installed system. (2 marks)

(b) Explain two factors that should be considered when choosing changeover strategy for a newly installed system. (4 marks)

(c) Alex, a systems analyst with a certain software company, developed several modules that would be joined into single system:

(i) Identify the most appropriate testing strategy that he could use justifying your answer. (2 marks)

(ii) Explain two benefits of using the testing strategy identified in (i). (4 marks)

(d) (i) Joachim a systems analyst with Day-Soft Ltd. was instructed to maintain documentation for a developed system. Explain two uses of documentation that could have influenced the instruction. (4 marks)

(ii) Mamboleo Institute of Technology intends to acquire a student’s online examination system. Explain ways in which the institute could use to achieve the objective. (4 marks)

Answer

(a) Types of Maintenance:

  • Corrective Maintenance: Fixing errors or defects discovered after implementation.
  • Adaptive Maintenance: Modifying the system to cope with changes in the environment (e.g., new OS, regulations).
  • Perfective Maintenance: Enhancing performance, usability, or maintainability even if the system is working correctly.
  • Preventive Maintenance: Making changes to reduce the likelihood of future failures.

(b) Factors for Choosing Changeover Strategy:

  • Risk Tolerance: How much risk (data loss, system failure, operational disruption) can the organization tolerate? (Direct=High Risk, Parallel/Pilot=Lower Risk).
  • System Criticality: How essential is the system to core business operations? Critical systems often favor lower-risk strategies like parallel or phased.
  • Cost and Resources: Parallel running is expensive; direct is cheapest but riskiest. Pilot/phased require careful planning and management.
  • Time Constraints: Direct changeover is fastest; parallel is slowest.

(c)(i) Testing Strategy:

  • Integration Testing. Justification: This strategy specifically focuses on testing the interfaces and interactions between different modules as they are combined into larger subsystems or the complete system.

(c)(ii) Benefits of Integration Testing:

  • Verifies Module Interaction: Ensures that independently developed modules work correctly together.
  • Early Defect Detection: Identifies interface problems and inter-module communication errors early before full system testing.
  • Builds Confidence: Gradually builds confidence in the combined system components.

(d)(i) Uses of Documentation for Maintenance:

  • Understanding the System: Provides maintainers with knowledge of the system's design, structure, and logic, enabling them to make changes effectively.
  • Impact Analysis: Helps assess how changes in one part of the system might affect other parts.
  • Training New Staff: Serves as a reference for new team members involved in maintenance.

(d)(ii) Acquiring Online Examination System:

  • In-house Development: Build the system using internal staff or resources.
  • Purchase Off-the-Shelf Software (COTS): Buy a pre-existing examination system package.
  • Customization of COTS: Purchase an off-the-shelf package and modify it to meet specific needs.
  • Outsourcing: Hire an external company to develop the system.
  • Cloud-based Service (SaaS): Subscribe to an online examination platform.

Question 6 (20 marks)

(a) (i) Outline two qualities of a good system document. (2 marks)

(ii) Outline three responsibilities of a project manager during system development. (3 marks)

(b) Explain three factors that could be considered when choosing system design tools. (6 marks)

(c) With the aid of an example in each case. describe each of the following type of relationships:

(i) one-to-one; (3 marks)

(ii) one-to-many; (3 marks)

(iii) many-to-many. (3 marks)

Answer

(a)(i) Qualities of Good System Document:

  • Clear and Understandable
  • Accurate and Up-to-date
  • Complete and Comprehensive
  • Well-organized and Accessible
  • Consistent

(a)(ii) Responsibilities of Project Manager:

  • Planning and Scheduling: Defining tasks, resources, and timelines.
  • Resource Management: Allocating and managing budget, personnel, and tools.
  • Risk Management: Identifying and mitigating potential project risks.
  • Communication: Facilitating communication between stakeholders, team members, and management.
  • Monitoring and Control: Tracking progress against the plan and taking corrective action.

(b) Factors for Choosing Design Tools:

  • Methodology Compatibility: Does the tool support the chosen development methodology (e.g., structured, OO, Agile)?
  • Functionality: Does the tool provide the necessary features for modeling (e.g., DFDs, ERDs, UML), code generation, documentation?
  • Team Familiarity/Skills: Is the development team already skilled in using the tool, or is training required?
  • Integration: Does the tool integrate well with other development tools (e.g., code repositories, testing tools)?
  • Cost and Licensing: Is the tool within the project budget?

(c) Types of Relationships (with examples):

  • (i) One-to-One (1:1): Each instance in one entity relates to exactly one instance in another entity, and vice versa. Example: A `User` entity might have a 1:1 relationship with a `UserProfile` entity (each user has one profile, each profile belongs to one user).
  • (ii) One-to-Many (1:M): One instance in the 'one' entity can relate to multiple instances in the 'many' entity, but each instance in the 'many' entity relates to only one instance in the 'one' entity. Example: A `Department` entity has a 1:M relationship with an `Employee` entity (one department has many employees, but each employee belongs to only one department).
  • (iii) Many-to-Many (M:N): One instance in the first entity can relate to multiple instances in the second entity, AND one instance in the second entity can relate to multiple instances in the first entity. Example: A `Student` entity has an M:N relationship with a `Course` entity (a student can enroll in many courses, and a course can have many students). This usually requires a linking/junction table in database design (e.g., `Enrollment` table).
