Question 1 (20 marks)

(a) (i) Outline two typical qualities of a systems analyst in organizations. (2 marks)

(ii) Distinguish between scientific and deterministic approaches as used in systems thinking. (4 marks)

(b) Explain three limitations of hard system thinking as used in development. (6 marks)

(c) (i) Outline four elements of a system design. (4 marks)

(ii) Alice intends to create system documentation for her company’s information system. Explain two uses of the documentation. (4 marks)

Answer

(a)(i) Qualities of Systems Analyst:

  • Analytical Skills: Ability to break down complex problems.
  • Communication Skills: Ability to interact effectively with users, management, and technical staff.
  • Problem-Solving Skills: Ability to identify issues and devise effective solutions.
  • Technical Knowledge: Understanding of IT concepts, hardware, software, and methodologies.

(a)(ii) Scientific vs Deterministic Approaches:

Scientific ApproachDeterministic Approach
Emphasizes observation, hypothesis testing, and empirical evidence to understand system behavior.Assumes system behavior is predictable and outcomes can be determined precisely based on inputs and rules.
Acknowledges uncertainty and uses methods like statistics.Focuses on cause-and-effect relationships with predictable outcomes.

(b) Limitations of Hard System Thinking:

  • Difficulty with Ill-Defined Problems: Struggles with complex social or organizational problems where objectives are unclear or conflicting.
  • Neglects Human Factors: Tends to overlook subjective perspectives, politics, culture, and human behavior within the system.
  • Oversimplification: May oversimplify complex realities to fit quantitative models, potentially missing crucial nuances.

(c)(i) Elements of System Design:

  • Output Design: Reports, screens, files needed by users.
  • Input Design: Forms, screens, methods for data entry.
  • Process Design: Logic, algorithms, procedures for transforming input to output.
  • Database/File Design: Structure and organization of stored data.
  • Control Design: Security, backup, recovery, audit trails.
  • Interface Design: How the system interacts with users and other systems.

(c)(ii) Uses of System Documentation:

  • System Understanding: Helps users, developers, and maintainers understand how the system works.
  • Maintenance and Enhancement: Provides essential information for modifying or upgrading the system.
  • Training: Serves as a basis for creating user training materials.
  • Communication: Facilitates communication among different stakeholders involved with the system.

Question 2 (20 marks)

(a) Explain the term robustness as used in information. (2 marks)

(b) Differentiate between input mask and validation text as used in data input. (4 marks)

(c) (i) Explain the term respondent as used in system investigation. (2 marks)

(ii) Maria was discussing benefits of usability as a quality in system design. Explain two benefits she could have emphasized. (4 marks)

(d) (i) Liz intends to select a file organization method for an information system. Outline four criteria that she is likely to consider. (4 marks)

(ii) Joy was required to formulate a term of reference document for a proposed information system. Outline four factors that she could consider to carry out the task. (4 marks)

Answer

(a) Robustness:

  • The ability of a system to handle errors during execution or incorrect/unexpected input gracefully, without crashing or producing catastrophic failures.

(b) Input Mask vs Validation Text:

Input MaskValidation Text
Defines a specific format for data entry (e.g., `(###) ###-####` for phone number). Guides the user during input.A message displayed to the user when entered data violates a validation rule (e.g., "Please enter a valid date"). Explains the error after input.
Controls the *format* of input.Provides feedback on *invalid* input based on rules.

(c)(i) Respondent:

  • An individual who provides data or answers questions during system investigation, typically through methods like interviews or questionnaires.

(c)(ii) Benefits of Usability:

  • Increased User Productivity/Efficiency: Users can complete tasks faster and with fewer errors.
  • Reduced Training Costs: An intuitive system requires less time and resources for user training.
  • Increased User Satisfaction: Users are happier and more likely to adopt and use a system that is easy and pleasant to interact with.
  • Fewer Errors: Good usability design can prevent common user mistakes.

(d)(i) Criteria for Selecting File Organization:

  • Volatility: How often is data added, deleted, or updated?
  • Activity: What proportion of records are processed in a typical run?
  • Access Requirements: Is sequential access sufficient, or is direct/random access needed frequently?
  • Storage Medium Characteristics: Capabilities and limitations of the storage device (e.g., tape vs. disk).
  • Response Time Requirements: How quickly must data be retrieved?

(d)(ii) Factors for Formulating Term of Reference (ToR):

  • Problem Definition: Clear understanding of the problem or opportunity the system addresses.
  • Project Objectives: Specific, measurable goals the project aims to achieve.
  • Project Scope: Defined boundaries of what is included and excluded.
  • Constraints: Known limitations (budget, time, resources, technology).
  • Stakeholder Identification: Knowing who needs to be involved or consulted.

Question 3 (20 marks)

(a) Differentiate between one-off costs and intangible costs as used systems analysis. (4 marks)

(b) Maro intends to use Entity Life History (ELH) model to analyze a proposed information system. Explain two advantages of the model. (4 marks)

(c) With the aid of a diagram in each case, describe the following types of relationships:

(i) one-to-many; (2 marks)

(ii) many-to-many. (2 marks)

(d) (i) Explain two weaknesses of Hierarchical Input Process-Output (HIPO) charts as used in systems design. (4 marks)

(ii) Jorum used phased changeover during implementation of an information system. Explain two possible drawbacks of the changeover strategy. (4 marks)

Answer

(a) One-off vs Intangible Costs:

One-off CostsIntangible Costs
Costs incurred only once during system development or acquisition (e.g., hardware purchase, initial software development).Costs that are difficult to quantify in monetary terms (e.g., loss of customer goodwill, reduced employee morale during change).
Usually capital expenditures.Often related to non-financial impacts.

(b) Advantages of Entity Life History (ELH):

  • Dynamic View: Models the sequence of events and state changes an entity undergoes, providing a dynamic perspective missing in static models like ERDs.
  • Process Validation: Helps verify that the system design correctly handles all possible events and state transitions for each entity.
  • Completeness Check: Ensures all valid actions and sequences in an entity's life are considered.

(c) Relationships (with Diagrams):

(i) One-to-Many (1:M): One instance of Entity A relates to zero, one, or many instances of Entity B. Each instance of Entity B relates to exactly one instance of Entity A.

+------------+ 1      M +------------+
| Department |----------| Employee   |
+------------+          +------------+
(A department has many employees)

(ii) Many-to-Many (M:N): One instance of Entity A relates to zero, one, or many instances of Entity B. One instance of Entity B relates to zero, one, or many instances of Entity A.

+---------+ M      N +--------+
| Student |----------| Course |
+---------+          +--------+
(A student takes many courses; a course has many students)
*Requires a linking table (e.g., Enrollment) in relational databases.

(d)(i) Weaknesses of HIPO Charts:

  • Limited Logic Representation: Does not clearly show decision logic (selection) or loops (iteration) within modules, unlike flowcharts or structured English.
  • Focus on Hierarchy: Emphasizes the hierarchical structure but may obscure the actual flow of control or data between modules.
  • Can Become Cumbersome: For large systems, maintaining a full set of HIPO diagrams (VTOC, Overview, Detail) can be laborious.

(d)(ii) Drawbacks of Phased Changeover:

  • Temporary Interfaces: May require building temporary interfaces between the new modules and the remaining parts of the old system, adding complexity and cost.
  • Extended Transition Period: The organization operates with a mix of old and new systems for a longer period, which can be confusing for users.
  • Potential Inconsistencies: Data inconsistencies might arise between the old and new system parts during the transition.

Question 4 (20 marks)

(a) (i) State two participants that could be invited for walkthrough meetings during systems development. (1 mark)

(ii) Outline three types of applications controls that could be incorporated in an information system. (3 marks)

(b) Cindy, an intern student with a certain system development company was required to develop a test plan for information. Outline four guidelines that she could use. (4 marks)

(c) With the aid of diagram in each case, describe two symbols that could be found in a program flowchart. (4 marks)

(d) County Coffee Society Ltd. intends to develop a system to automate its operations. When a farmer delivers the coffee harvest to designated coffee milling station, the farmer’s details are retrieved from farmers file for verification purposes. The farmer then proceeds to put his/her coffee bags on the weighing machine. The attendant then updates the farmer’s delivery file and copy receipt is issued to farmer. The farmer may then proceed to the store to take farm inputs and foods. At the store, the attendant updates the borrowers file and generates an invoice. At the end month, the accountant of the milling station is required to process farmer’s payment and record on the farmer’s debit file transmit the money to the farmer’s bank account.

Draw a dataflow diagram to represent the system. (8 marks)

Answer

(a)(i) Walkthrough Participants:

  • Author (Developer/Analyst of the work product)
  • Moderator/Chairperson
  • Reviewers (Peers, e.g., other developers, analysts, testers)
  • Scribe/Recorder
  • User Representative (Optional, depending on the item reviewed)

(a)(ii) Application Controls:

  • Input Controls: (e.g., Validation rules, completeness checks, reasonableness checks)
  • Processing Controls: (e.g., Run-to-run totals, transaction logging, error handling routines)
  • Output Controls: (e.g., Report reconciliation, distribution lists, output verification)
  • Data File Controls: (e.g., Access restrictions, backup/recovery procedures, file labeling)

(b) Test Plan Guidelines:

  • Define Scope and Objectives: Clearly state what will and will not be tested and the goals of testing.
  • Identify Test Items/Features: List the specific modules, functions, or requirements to be tested.
  • Specify Test Approach: Describe the types of testing (unit, integration, system, UAT) and techniques to be used.
  • Define Pass/Fail Criteria: Establish clear criteria for determining whether a test case passes or fails.
  • Outline Test Environment: Specify hardware, software, and data requirements for testing.
  • Schedule and Resources: Estimate time, personnel, and tools needed for testing activities.

Question 5 (20 marks)

(a) (i) Outline two characteristics of graphical user interface. (2 marks)

(ii) Differentiate between verification and validation as used in waterfall model. (4 marks)

(b) Aaron was preparing for a presentation on reasons that necessitates system maintenance. Outline four reasons that he is likely to have listed on his presentation. (4 marks)

(c) Dan used black-box testing on an information system that he was developing. Outline four limitations that he could experience. (4 marks)

(d) Raph intends to use prototyping for determining user requirements only.

(i) Explain the most appropriate type of prototype he could have used. (2 marks)

(ii) Explain two advantages of the type of prototyping in (i). (4 marks)

Answer

(a)(i) Characteristics of GUI:

  • Visual Components: Uses windows, icons, menus, pointers (WIMP).
  • Direct Manipulation: Users interact directly with visual objects (e.g., drag and drop).
  • Event-Driven: Actions are typically triggered by user events (mouse clicks, key presses).
  • User-Friendly: Generally easier to learn and use than command-line interfaces.

(a)(ii) Verification vs Validation (Waterfall):

VerificationValidation
Checks if the product of a phase meets the requirements specified in the *previous* phase. "Are we building the product right?"Checks if the final system meets the *original* user requirements specified at the beginning. "Are we building the right product?"
Typically done at the end of each phase (e.g., design review checks against analysis specs).Typically done at the end of the project (e.g., User Acceptance Testing).

(b) Reasons Necessitating System Maintenance:

  • Correcting Errors (Bugs): Fixing defects discovered after deployment.
  • Adapting to Environment Changes: Modifying the system due to new hardware, OS updates, or changes in regulations.
  • Enhancing Functionality: Adding new features or improving existing ones based on user requests (Perfective).
  • Improving Performance/Efficiency: Optimizing code or database structures (Perfective).
  • Preventing Future Problems: Making changes to improve reliability or maintainability (Preventive).

(c) Limitations of Black-Box Testing:

  • Limited Code Coverage: Cannot guarantee all code paths are tested as internal structure is unknown.
  • Inefficient Test Case Design: May lead to redundant test cases or missing critical scenarios without knowledge of internal logic.
  • Difficult to Isolate Faults: When a test fails, it's harder to pinpoint the exact location of the defect in the code.
  • Cannot Detect Certain Error Types: May miss internal logic errors or inefficiencies that don't manifest as incorrect output for the chosen inputs.

(d)(i) Appropriate Prototype Type:

  • Throwaway Prototyping (also known as Rapid Prototyping or Mockups). Justification: The primary goal is solely requirements determination and clarification. The prototype is built quickly to get user feedback and is then discarded; it does not become part of the final system.

(d)(ii) Advantages of Throwaway Prototyping:

  • Clarifies Requirements Effectively: Users can see and react to a visual model, helping to refine and confirm requirements early.
  • Reduces Misunderstandings: Minimizes the risk of building the wrong system by getting early user validation.
  • Faster Feedback Cycle: Allows for quick iterations based on user input during the analysis phase.
  • Low Investment: Since it's discarded, less effort is spent on making it robust or fully functional compared to evolutionary prototypes.

Question 6 (20 marks)

(a) (i) Outline three benefits of carrying out systems analysis in organizations. (3 marks)

(ii) Differentiate between probabilistic and adaptive systems. (4 marks)

(b) Explain three circumstances that would necessitate use of decision support system (DSS) in organizations. (6 marks)

(c) (i) Explain the term environment as used in information systems. (2 marks)

(ii) Jose intends to carry-out an on job training for an information system that he was developing for one of his clients. Outline five methods that he is likely to use. (5 marks)

Answer

(a)(i) Benefits of Systems Analysis:

  • Clear Problem Understanding: Ensures the real problem is identified, not just symptoms.
  • Improved Requirements Definition: Leads to a more accurate and complete specification of what the system should do.
  • Better System Design: Provides a solid foundation for designing an effective and efficient solution.
  • Reduced Development Risk: Identifies potential issues and feasibility constraints early.

(a)(ii) Probabilistic vs Adaptive Systems:

Probabilistic SystemAdaptive System
Its behavior or output cannot be precisely predicted for a given input, involving an element of randomness or uncertainty.Can modify its own behavior or structure in response to changes in its environment or its own performance.
Focuses on likelihoods and statistical outcomes.Focuses on learning and self-adjustment.
Example: A weather forecasting system.Example: A spam filter that learns from user feedback.

(b) Circumstances Necessitating DSS:

  • Semi-structured or Unstructured Decisions: Problems where judgment, evaluation, and insight are required, and procedures are not fully predefined (e.g., new product introduction, market analysis).
  • Need for Data Analysis and Modeling: Situations requiring analysis of large data volumes, "what-if" scenarios, or complex modeling (e.g., financial planning, sales forecasting).
  • Supporting Managerial Decision Making: Assisting middle and upper management with non-routine decisions requiring more than simple transaction processing.

(c)(i) Environment (Information Systems):

  • Everything outside the system's boundary that interacts with or influences the system. This includes users, other systems, organizational factors, regulations, competitors, etc.

(c)(ii) On-the-Job Training Methods:

  • Coaching/Mentoring: Experienced user guides the trainee directly.
  • Job Rotation: Trainee moves through different roles to learn various system aspects.
  • Demonstration: Trainer shows how to perform tasks on the live system.
  • Step-by-Step Instructions/Checklists: Providing guided procedures for specific tasks.
  • Shadowing: Trainee observes an experienced user performing tasks.

Question 7 (20 marks)

(a) (i) Outline three logical security measures that could be incorporated in a system. (3 marks)

(ii) Flo was required to review a user manual document that was rejected by system users. Assume you were consulted by Flo, advice her on four qualities that she could consider on the document. (4 marks)

(b) Fred intends to use state transition diagram to design a proposed information system. Explain two benefits that he would realize from using the preferred design tool. (4 marks)

(c) Describe each of the following SDLC models:

(i) iterative; (2 marks)

(ii) spiral. (2 marks)

(d) Angie intends to design a questionnaire for collecting data during feasibility studies. Outline five guidelines that she could use to achieve her goal. (5 marks)

Answer

(a)(i) Logical Security Measures:

  • User Authentication (Passwords, Biometrics, Multi-Factor Authentication).
  • Access Control Lists (ACLs) / Role-Based Access Control (RBAC).
  • Data Encryption (at rest and in transit).
  • Audit Trails / Activity Logging.
  • Firewalls / Intrusion Detection Systems (IDS).

(a)(ii) Qualities for User Manual Review:

  • Clarity: Is the language clear, concise, and easy for the target user to understand? Avoid jargon.
  • Accuracy: Does the manual correctly describe how the system functions? Are the steps accurate?
  • Completeness: Does it cover all necessary functions and common tasks? Are there any obvious omissions?
  • Organization & Navigability: Is the manual well-structured (e.g., table of contents, index)? Can users easily find the information they need?
  • Usefulness of Examples/Diagrams: Are examples relevant and diagrams clear?

(b) Benefits of State Transition Diagrams:

  • Modeling System Behavior: Clearly shows the different states an object or system can be in and the events that trigger transitions between states.
  • Identifying Valid/Invalid Transitions: Helps ensure all possible state changes are considered and handled correctly, including error states.
  • Communication Tool: Provides a visual way to communicate complex system behavior to developers and stakeholders.

(c) SDLC Models:

  • (i) Iterative Model: Develops the system in repeated cycles (iterations). Each iteration builds upon the previous one, adding more functionality until the full system is complete. Allows for feedback and refinement during the process.
  • (ii) Spiral Model: An iterative, risk-driven model. Each cycle (spiral) involves planning, risk analysis, engineering (development/testing), and evaluation. Suitable for large, complex, high-risk projects.

(d) Questionnaire Design Guidelines:

  • Define Clear Objectives: Know exactly what information you need to gather.
  • Keep it Concise: Ask only necessary questions; avoid excessive length.
  • Use Clear and Unambiguous Language: Avoid jargon, acronyms, or leading questions.
  • Logical Flow: Group related questions together and order them logically.
  • Use Appropriate Question Types: Mix closed and open questions effectively based on the data needed.
  • Pilot Test: Test the questionnaire on a small group before wider distribution to identify confusing questions or issues.
  • Ensure Anonymity/Confidentiality (if applicable): Reassure respondents their answers will be handled appropriately.

Question 8 (20 marks)

(a) Explain each of the following terms as used in system design:

(i) modularity; (2 marks)

(ii) cohesion. (2 marks)

(b) A group of students were carrying out an assignment on characteristics of an information system. Outline four characteristics that they are likely to have listed. (4 marks)

(c) (i) Explain the term selection bias as used in feasibility study. (2 marks)

(ii) With the aid of a diagram, describe a closed-loop system. (4 marks)

(d) Jemo was required to review an information system in order for it to accommodate a new technology.

(i) Identify the most appropriate type of maintenance that he could have used justifying your answer. (2 marks)

(ii) Explain two challenges he could have experience while carrying out the maintenance identified in (i). (4 marks)

Answer

(a) System Design Terms:

  • (i) Modularity: Designing a system as a collection of independent, interchangeable components (modules), each responsible for a specific function. Promotes reusability and maintainability.
  • (ii) Cohesion: The degree to which the elements within a single module belong together and focus on a single task. High cohesion is desirable (module does one thing well).

(b) Characteristics of an Information System:

  • Components: Composed of hardware, software, data, people, processes.
  • Purpose: Designed to achieve specific organizational objectives.
  • Input, Processing, Output: Takes input data, processes it, and produces output information.
  • Feedback: Often includes feedback mechanisms for control and improvement.
  • Boundary & Environment: Has defined boundaries separating it from its environment, with which it may interact.

(c)(i) Selection Bias (Feasibility Study):

  • Occurs when the sample or data chosen for the study is not representative of the overall situation, leading to skewed or inaccurate conclusions about feasibility. For example, only interviewing users who are enthusiastic about a new system.

(c)(ii) Closed-Loop System:

  • A system that uses feedback from its own output to control or modify its actions, aiming to maintain a desired state. It automatically adjusts based on the difference between the actual output and the desired output.
+---------+      +-----------+      +--------+
| Input   |----->| Processor |----->| Output |
+---------+      +-----------+      +--------+
    ^                 |                 |
    |                 |                 |
    | Control Signal  |                 | Feedback
    +-----------------| Sensor/Monitor  |<----------+
                      +-----------------+

Description: Input enters the processor, producing output. A sensor monitors the output and provides feedback. The processor compares feedback to a desired state and sends a control signal to adjust its processing or the input.

(d)(i) Type of Maintenance:

  • Adaptive Maintenance. Justification: This type involves modifying the system to cope with changes in the technical environment, such as accommodating new technology.

(d)(ii) Challenges of Adaptive Maintenance:

  • Integration Issues: Ensuring the new technology integrates smoothly with existing system components.
  • Testing Complexity: Thorough testing is required to ensure the changes haven't negatively impacted other parts of the system (regression testing).
  • Learning Curve: The maintenance team may need to learn the new technology.
  • Cost and Time: Adapting systems to new technologies can be resource-intensive.
