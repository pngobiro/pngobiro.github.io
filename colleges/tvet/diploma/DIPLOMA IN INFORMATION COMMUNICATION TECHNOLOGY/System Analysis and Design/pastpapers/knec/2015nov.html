Question 1 (20 marks)

(a) Outline four roles of a systems analyst as a change agent. (4 marks)

(b) Explain two characteristics of deterministic systems. (4 marks)

(c) Distinguish between waterfall and phased approaches as used in system development. (4 marks)

(d) A well-designed user documentation would yield better results when used in an organization. Explain four good qualities that could be in this documentation. (8 marks)

Answer

(a) Roles of Systems Analyst as Change Agent:

  • Catalyst: Initiates and motivates change within the organization.
  • Problem Solver: Identifies issues and proposes system-based solutions.
  • Mediator: Facilitates communication and resolves conflicts between users, management, and technical teams.
  • Trainer/Educator: Helps users understand and adapt to the new system.

(b) Characteristics of Deterministic Systems:

  • Predictable Output: For a given input and initial state, the output can be precisely determined.
  • Known Interactions: The interactions between system components follow fixed rules.
  • No Randomness: Behavior does not involve chance or probability.

(c) Waterfall vs Phased Approaches:

WaterfallPhased
Linear, sequential progression through distinct SDLC phases (Analysis -> Design -> Code -> Test -> Deploy).System is developed and implemented in functional stages or versions (phases). Each phase might follow a mini-waterfall or iterative approach.
Entire system delivered at the end.Delivers usable parts of the system incrementally.
Less flexible to changes once a phase is complete.More flexible; allows feedback from early phases to influence later ones.

(d) Qualities of Good User Documentation:

  • Clarity: Easy to understand, using clear language appropriate for the target audience, avoiding jargon.
  • Accuracy: Correctly reflects how the system actually works.
  • Completeness: Covers all necessary features, tasks, and potential issues users might encounter.
  • Accessibility/Organization: Well-structured with indexes, tables of contents, search functions, making information easy to find.
  • Conciseness: Provides necessary information without unnecessary detail.

Question 2 (20 marks)

(a) Define each of the following terms as used in system analysis and design:

(i) sub-system; (2 marks)

(ii) environment. (2 marks)

(b) Explain two economic challenges experienced during systems development. (4 marks)

(c) Differentiate between State Transition Diagram (STD) and Entity Life History (ELH) as used in system design. (4 marks)

(d) In the design of a user interface for an information system, certain input considerations should be considered. Explain four of these considerations. (8 marks)

Answer

(a) Definitions:

  • (i) Sub-system: A smaller system that is part of a larger system. It functions independently to some extent but also interacts with other sub-systems to achieve the overall goal of the larger system.
  • (ii) Environment: Everything outside the defined boundary of a system that interacts with or influences the system.

(b) Economic Challenges in Systems Development:

  • Budget Overruns: Difficulty in accurately estimating costs, leading to exceeding the allocated budget.
  • Quantifying Benefits: Difficulty in measuring intangible benefits (like improved morale or customer satisfaction) to justify the system's cost.
  • Resource Allocation: Competing demands for limited financial and human resources within the organization.

(c) STD vs ELH:

State Transition Diagram (STD)Entity Life History (ELH)
Models the different states an *entire system* or a *complex object* can be in and the events that cause transitions between these states.Models the sequence of events (actions) that affect a *single entity type* throughout its lifetime, from creation to deletion.
Focuses on system-wide or object behavior and control flow.Focuses on the valid sequence of operations for one specific data entity.
Often used for real-time systems or complex object modeling.Often used in data-centric methodologies (like SSADM) to validate process models against data structures.

(d) User Interface Input Considerations:

  • Consistency: Use consistent terminology, layouts, and controls throughout the interface.
  • Clarity: Ensure labels, instructions, and prompts are clear and unambiguous.
  • Efficiency: Minimize user effort (e.g., keystrokes, mouse clicks) required for data entry. Use defaults where appropriate.
  • Feedback: Provide immediate feedback for user actions (e.g., confirmation messages, error alerts).
  • Error Prevention/Handling: Design inputs to prevent errors (e.g., using dropdowns instead of free text) and provide helpful error messages if they occur.
  • Flexibility: Allow users some control over the interface (e.g., shortcuts, customization options).

Question 3 (20 marks)

(a) Outline four security concerns that could be addressed in the design of an information system. (4 marks)

(b) Explain two circumstances under which soft system thinking could be used in an organization. (4 marks)

(c) Differentiate between HIPO charts and IPO charts as used in information systems. (4 marks)

(d) Figure 1 shows a closed loop control system used in information systems. Explain the functions of the parts labelled (i), (ii), (iii) and (iv). (8 marks)

Answer

(a) Security Concerns in Design:

  • Confidentiality: Preventing unauthorized disclosure of sensitive information.
  • Integrity: Ensuring data accuracy and preventing unauthorized modification.
  • Availability: Ensuring the system and its data are accessible to authorized users when needed.
  • Authentication: Verifying the identity of users accessing the system.
  • Authorization: Granting appropriate access levels and permissions to authenticated users.

(b) Circumstances for Soft System Thinking:

  • Ill-Defined Problems: Situations where the problem itself is unclear, complex, and involves multiple conflicting perspectives (often involving human activity systems).
  • Organizational Change: Understanding the social and political dynamics when planning or implementing significant organizational changes involving people and processes.
  • Exploring Different Viewpoints: When it's crucial to understand and accommodate diverse stakeholder worldviews and objectives.

(c) HIPO vs IPO Charts:

HIPO (Hierarchical Input-Process-Output)IPO (Input-Process-Output)
A set of diagrams showing system hierarchy (VTOC) and module details (Overview/Detail IPO diagrams).A simple diagram showing the inputs, main processing steps, and outputs for a *single* module or program.
Focuses on functional decomposition and modular structure.Focuses on the basic transformation within one specific process.
More comprehensive, includes hierarchy chart.Simpler, focuses only on one module's I-P-O.

(d) Closed Loop System Components (Fig 1):

  • (i) Comparator/Error Detector: Compares the reference input (desired state) with the feedback signal (actual state) to generate an error signal indicating the difference.
  • (ii) Processor/Controller: Receives the error signal and determines the necessary action or adjustment to the system/process to reduce the error.
  • (iii) Sensor/Monitor: Measures the actual output of the system.
  • (iv) Feedback Path: Transmits the measured output signal from the sensor back to the comparator.

Question 4 (20 marks)

(a) Outline four limitation of a prototyping as used in system design. (4 marks)

(b) Explain two disadvantages of direct changeover technique as used in systems implementation. (4 marks)

(c) With the aid of sketches, explain two types of relationships used in ER diagrams. (4 marks)

(d) The Strategic Plan Steering Committee of Hazel Ltd. intends to acquire a new information system for the organization. Explain four factors that should be considered when choosing the system. (8 marks)

Answer

(a) Limitations of Prototyping:

  • "Code and Fix" Mentality: Can lead to less structured design and poor documentation if not managed properly.
  • User Misconceptions: Users might perceive the prototype as the final system, leading to unrealistic expectations about delivery time or robustness.
  • Incomplete Analysis: Focus on the prototype might lead to overlooking broader system requirements or integration issues.
  • Can Be Time-Consuming: Building and refining prototypes, especially evolutionary ones, can take significant time.

(b) Disadvantages of Direct Changeover:

  • High Risk: If the new system fails, there is no immediate fallback to the old system, potentially causing major operational disruption.
  • Insufficient Testing: The pressure for a quick switch might lead to inadequate testing before going live.
  • User Stress: Requires users to adapt immediately to the new system without a transition period.

(c) ER Diagram Relationships (Sketches):

  • One-to-Many (1:M): 
    +---+ 1      M +---+
| A |---|------| B |
+---+          +---+
    (One A relates to Many B)
  • Many-to-Many (M:N): 
    +---+ M      N +---+
| A |----------| B |
+---+          +---+
    (Many A relate to Many B)
  • One-to-One (1:1): 
    +---+ 1      1 +---+
| A |----------| B |
+---+          +---+
    (One A relates to One B)

(d) Factors for Choosing/Acquiring a New System:

  • Functional Requirements: Does the system meet the organization's specific operational and business needs?
  • Cost: Includes purchase/development cost, implementation cost, training cost, ongoing maintenance cost (Total Cost of Ownership - TCO).
  • Technical Fit: Is the system compatible with the existing IT infrastructure (hardware, OS, network, database)?
  • Vendor Support & Viability: If buying, is the vendor reputable, financially stable, and able to provide adequate long-term support and updates?
  • Usability: Is the system easy for end-users to learn and operate effectively?
  • Scalability & Flexibility: Can the system grow with the organization and adapt to future needs?
  • Security: Does the system meet the organization's security requirements?

Question 5 (20 marks)

(a) Outline four limitations of telephone interview as a data collection tool. (4 marks)

(b) Explain two issues addressed by the operational feasibility study of an information system. (4 marks)

(c) Distinguish between black box testing and white box testing as used in systems development. (4 marks)

(d) With the aid of a diagram, outline the stages of Soft System Methodology. (8 marks)

Answer

(a) Limitations of Telephone Interview:

  • Lack of Visual Cues: Cannot observe respondent's body language or reactions.
  • Limited Rapport: Harder to build trust and rapport compared to face-to-face interviews.
  • Difficulty with Complex/Sensitive Questions: May not be suitable for discussing complex topics or sensitive personal information.
  • Potential for Distractions: Respondent might be distracted by their environment.
  • Limited Duration: Respondents may be less willing to engage in long interviews over the phone.

(b) Issues Addressed by Operational Feasibility:

  • User Acceptance: Will end-users accept and use the new system? Are there potential resistance issues?
  • Organizational Impact: How will the system affect current business processes, organizational structure, and job roles?
  • Training Needs: What level of training will be required for users and technical staff?
  • Resource Availability: Are the necessary personnel (operators, support staff) available to run the system effectively?

(c) Black Box vs White Box Testing:

Black Box TestingWhite Box Testing
Testing based on input/output behavior without knowledge of internal code structure.Testing based on knowledge of the internal code structure, logic paths, and conditions.
Focuses on functional requirements ("Does it do what it's supposed to?").Focuses on code coverage and internal logic ("Does the code work correctly?").

(d) Soft System Methodology (SSM) Stages (Diagram Description):

SSM is typically represented as a cycle with stages moving between the 'Real World' and 'Systems Thinking World':

  1. Real World Problem Situation: Unstructured problem perceived.
  2. Problem Situation Expressed: Rich picture created, key elements identified.
  3. Root Definitions: Define relevant systems based on CATWOE (Customers, Actors, Transformation, Weltanschauung, Owners, Environment).
  4. Conceptual Models: Build models of the systems defined in Root Definitions (what the system *must* do).
  5. Comparison: Compare Conceptual Models (4) with the Expressed Problem Situation (2).
  6. Debate/Define Changes: Discuss feasible and desirable changes based on the comparison.
  7. Take Action: Implement agreed changes in the Real World.

(A diagram would show these stages, often with arrows indicating flow and iteration, distinguishing between real-world actions and systems thinking activities).

Question 6 (20 marks)

(a) Explain two benefits that could be derived from using an adaptive information system. (4 marks)

(b) Distinguish between Jackson Structured Programming (JSP) and Jackson System Development (JSD) as used in systems development. (4 marks)

(c) Jewel Company Ltd. intends to train its employees in order to improve the overall performance of the organization. Explain two types of users that could be trained. (4 marks)

(d) Consider a project with initial costs of Kes 100,000 which produces an income of Kes 500,000 p.a. for five years. Assuming a discount rate of 10%, determine the net present value of the project. (8 marks)

Answer

(a) Benefits of Adaptive Information System:

  • Improved Performance Over Time: Can learn from data or user interactions to optimize its responses or actions.
  • Resilience to Change: Can adjust its behavior or structure to cope with changes in its operating environment without requiring manual reprogramming.
  • Personalization: Can tailor its responses or interface to individual user preferences or needs.

(b) JSP vs JSD:

Jackson Structured Programming (JSP)Jackson System Development (JSD)
A method for designing the structure of *individual programs* based on the structure of the input and output data streams.A broader system development methodology that models the *real world* first (entities and actions) and then derives the system implementation from that model.
Focuses on program logic and data structures.Focuses on modeling the time-ordered behavior of real-world entities.
Produces program structure diagrams.Produces system specification diagrams (networks) based on entity life histories.

(c) Types of Users to be Trained:

  • End-Users/Operational Staff: Need training on how to use the system's features to perform their specific job tasks efficiently and correctly.
  • Technical Support/System Administrators: Need training on system installation, configuration, troubleshooting, maintenance, and security procedures.
  • Managers: May need training on how to interpret system reports, use decision support features, or oversee system usage.

(d) Net Present Value (NPV) Calculation:

Initial Cost = -100,000
Annual Income = 500,000
Period = 5 years
Discount Rate = 10% (0.10)

PV Factor (Annuity) for 5 years at 10% = [1 - (1 + r)^-n] / r
= [1 - (1.10)^-5] / 0.10
= [1 - 0.62092] / 0.10
= 0.37908 / 0.10
= 3.7908

Total PV of Income = Annual Income * PV Factor
= 500,000 * 3.7908
= 1,895,400

NPV = Total PV of Income - Initial Cost
= 1,895,400 - 100,000
= 1,795,400

The Net Present Value (NPV) of the project is Kes 1,795,400.

Question 7 (20 marks)

(a) (i) Outline three logical security measures that could be incorporated in a system. (3 marks)

(ii) Flo was required to review a user manual document that was rejected by system users. Assume you were consulted by Flo, advice her on four qualities that she could consider on the document. (4 marks)

(b) Fred intends to use state transition diagram to design a proposed information system. Explain two benefits that he would realize from using the preferred design tool. (4 marks)

(c) Describe each of the following SDLC models:

(i) iterative; (2 marks)

(ii) spiral. (2 marks)

(d) Angie intends to design a questionnaire for collecting data during feasibility studies. Outline five guidelines that she could use to achieve her goal. (5 marks)

Answer

(a)(i) Logical Security Measures:

  • User Authentication (Passwords, Biometrics, MFA).
  • Access Control Lists (ACLs) / Role-Based Access Control (RBAC).
  • Data Encryption.
  • Audit Trails.
  • Firewalls.

(a)(ii) Qualities for User Manual Review:

  • Clarity & Understandability
  • Accuracy
  • Completeness
  • Organization & Navigability
  • Relevance to Target Audience

(b) Benefits of State Transition Diagrams:

  • Modeling System Behavior: Clearly shows states and transitions triggered by events.
  • Identifying Valid/Invalid Transitions: Ensures all state changes are handled.
  • Communication Tool: Visually communicates complex behavior.

(c) SDLC Models:

  • (i) Iterative Model: Develops system in cycles, each adding functionality. Allows feedback.
  • (ii) Spiral Model: Iterative, risk-driven. Each cycle includes planning, risk analysis, engineering, evaluation. Good for complex, high-risk projects.

(d) Questionnaire Design Guidelines:

  • Define Clear Objectives.
  • Keep it Concise.
  • Use Clear and Unambiguous Language.
  • Logical Flow.
  • Use Appropriate Question Types.
  • Pilot Test.
  • Ensure Anonymity/Confidentiality (if applicable).

(a) Distinguish between usability and readability as used in systems design. (4 marks)

(b) (i) Explain the term prototype as used in system design. (2 marks)

(ii) Outline three examples of stakeholders that could be involved when carrying out a system project in organizations. (3 marks)

(i) limited entry; (2 marks)

(ii) extended entry; (2 marks)

(iii) mixed entry. (2 marks)

(i) chance/event; (2 marks)

(ii) terminal/edge; (2 marks)

(iii) decision. (2 marks)

Answer

(a) Usability vs Readability:

UsabilityReadability
The ease with which users can learn, use, and achieve goals effectively and efficiently with a system interface.The ease with which human readers can comprehend source code or documentation.
Focuses on user interaction and task completion.Focuses on clarity and understandability of the code/text itself.
Measured by task success rate, time on task, user satisfaction.Measured by code complexity metrics, adherence to style guides, clarity of comments/prose.

(b)(i) Prototype:

  • A working model or preliminary version of a system or component, built quickly to test concepts, gather user feedback, or demonstrate functionality before full development.

(b)(ii) Examples of Stakeholders:

  • End Users
  • Management/Sponsors
  • Development Team
  • Customers
  • Suppliers

(c) Types of Decision Table:

  • (i) Limited Entry: Conditions are Y/N/-, Actions are X/Blank.
  • (ii) Extended Entry: Conditions use values/ranges, Actions specify values.
  • (iii) Mixed Entry: Combines Limited and Extended entry rows.
