Question 1 (20 marks)

(a) (i) Explain the cause of scope creep during System Development. (2 marks)

(ii) Outline four advantages of an expert system. (4 marks)

(b) Differentiate between structured and unstructured decisions as used in systems theory. (4 marks)

(c) Describe two data elements that could be included in a data dictionary. (4 marks)

(d) Mara intends to design input and output interface for an Information System. Describe three types of interfaces she could use. (6 marks)

Answer

(a) (i) Cause of Scope Creep:

  • Uncontrolled changes or additions to project requirements after the initial scope has been defined and agreed upon, often due to unclear initial requirements, stakeholder requests, or poor change management.

(ii) Advantages of an Expert System:

  1. Consistency: Provides consistent answers/solutions for repetitive decisions.
  2. Expertise Preservation: Captures and retains knowledge of human experts.
  3. Accessibility: Makes expert-level knowledge available to non-experts or in remote locations.
  4. Speed: Can process information and reach conclusions faster than humans in specific domains.

(b) Structured vs. Unstructured Decisions:

Structured Decisions Unstructured Decisions
Repetitive and routine, with clear procedures for handling. Novel, non-routine, complex, with no predefined procedures.
Based on established rules and data. Require judgment, intuition, and evaluation of multiple factors.
Often made at lower operational levels. Typically made at higher strategic levels.
Example: Inventory reordering when stock falls below a threshold. Example: Deciding whether to enter a new market.

(c) Data Elements in a Data Dictionary:

  1. Data Element Name: A unique identifier for the data element (e.g., `CustomerID`, `OrderDate`).
  2. Description/Definition: A clear explanation of what the data element represents (e.g., "Unique identifier for a customer", "Date the order was placed").
  3. Data Type & Format: Specifies the type of data (e.g., numeric, text, date) and its format (e.g., YYYY-MM-DD, Integer, Varchar(50)).
  4. Validation Rules: Constraints or rules the data must adhere to (e.g., range checks, required field).
*(Note: Only two were required)*

(d) Types of Input/Output Interfaces:

  1. Graphical User Interface (GUI):
    • Uses visual elements like windows, icons, menus, and pointers (WIMP).
    • Allows interaction via mouse clicks, drag-and-drop, buttons, forms.
    • Example: Standard desktop applications (Windows, macOS).
  2. Command-Line Interface (CLI):
    • Text-based interface where users type commands to interact with the system.
    • Requires knowledge of specific commands.
    • Example: Linux terminal, Windows Command Prompt.
  3. Menu-Driven Interface:
    • Presents users with a list of options (menu) to choose from.
    • Simpler than CLI, often used in ATMs or older systems.
    • Navigation is done by selecting menu items.
  4. Natural Language Interface:
    • Allows users to interact using spoken or typed human language.
    • Requires sophisticated processing (NLP).
    • Example: Voice assistants (Siri, Alexa), chatbots.
*(Note: Only three were required)*

Question 2 (20 marks)

(a) Outline two examples of external problem indicators in the development of an Information System. (2 marks)

(b) Describe each of the following System Development methodologies:

(i) structured; (2 marks)

(ii) object-oriented. (2 marks)

(c) (i) Marcus prepared a requirement specification document for a proposed system. Explain two importance of this document during system development. (4 marks)

(ii) Differentiate between critical and dummy activities as used in ICT project management. (4 marks)

(d) A company hired a system analyst to train staff on the use of a new Information System. Describe three methods of training that he could use. (6 marks)

Answer

(a) External Problem Indicators for IS Development:

  1. Customer Complaints: Increased complaints about service speed, errors, or lack of features.
  2. Competitor Actions: Competitors launching new systems or services that offer a better experience or efficiency.
  3. Regulatory Changes: New laws or regulations requiring changes in data handling, reporting, or business processes.
*(Note: Only two were required)*

(b) System Development Methodologies:

(i) Structured Methodology:

  • Traditional, sequential approach (like Waterfall).
  • Emphasizes distinct phases (analysis, design, implementation, testing).
  • Focuses on processes and data flows, using tools like DFDs and ERDs.

(ii) Object-Oriented Methodology:

  • Views a system as a collection of interacting objects.
  • Combines data and processes (methods) within objects.
  • Uses concepts like encapsulation, inheritance, polymorphism, often with iterative development. Tools include UML diagrams (Use Case, Class diagrams).

(c) (i) Importance of Requirement Specification Document:

  1. Baseline for Design & Development: Provides a clear, agreed-upon foundation defining what the system must do, guiding subsequent design and coding phases.
  2. Basis for Testing & Validation: Serves as the criteria against which the final system is tested to ensure it meets the specified needs and functions correctly.
  3. Communication Tool: Facilitates understanding and agreement between stakeholders (users, developers, managers) about the system's scope and functionality.
*(Note: Only two were required)*

(ii) Critical vs. Dummy Activities:

Critical Activity Dummy Activity
An activity on the critical path of a project network diagram. An activity with zero duration used to show dependencies or maintain logic in a network diagram.
Has zero float/slack; any delay in this activity delays the entire project. Consumes no time or resources.
Represents actual work that needs to be done. Represents only a precedence relationship, often shown as a dashed line.

(d) Methods of Training Staff:

  1. Instructor-Led Classroom Training: Formal sessions led by the analyst or a trainer, covering system features and procedures, often with hands-on practice. Good for groups.
  2. One-on-One Coaching/Mentoring: Personalized training where the analyst works directly with individual users, tailored to their specific roles and pace.
  3. Workshops/Simulations: Interactive sessions where users work through realistic scenarios using the new system in a controlled environment.
  4. E-Learning/Computer-Based Training (CBT): Providing online modules, tutorials, or videos that staff can access and complete at their own pace.
*(Note: Only three were required)*

Question 3 (20 marks)

(a) Outline four roles of a project manager in ICT project management. (4 marks)

(b) (i) Explain the term system thinking. (2 marks)

(ii) A system analyst was reviewing an Information System at design stage. Explain two characteristics that this design should possess. (4 marks)

(c) Distinguish between straight and phased file conversion methods as used in system implementation. (4 marks)

(d) A supermarket has a reward system for its loyal customers. If a customer purchases goods worth shs. 30,000 or more, and the loyalty card has more than 3,000 points, the customer is awarded a voucher worth shs. 5,000 or a voucher worth shs.2,000 if the card has less than 3,000 points. The transaction is stored in a file and a report generated. Draw a system flowchart to represent this logic. (6 marks)

Answer

(a) Roles of a Project Manager:

  1. Planning: Defining scope, objectives, tasks, timelines, and resources.
  2. Organizing: Assembling and managing the project team and resources.
  3. Leading/Directing: Motivating the team, facilitating communication, resolving conflicts.
  4. Controlling/Monitoring: Tracking progress, managing risks, ensuring quality, managing budget and scope changes.

(b) (i) System Thinking:

  • A holistic approach to analysis that focuses on understanding how a system's constituent parts interrelate and how systems work over time and within the context of larger systems. It emphasizes seeing the whole picture, not just isolated components.

(ii) Characteristics of Good System Design:

  1. Modularity: The design should be broken down into independent, manageable components (modules) with well-defined interfaces. This improves maintainability and allows parallel development.
  2. Reliability: The system should perform consistently and correctly under expected conditions, with mechanisms for error handling and recovery.
  3. Efficiency: The system should use resources (CPU, memory, network) effectively and provide acceptable response times.
  4. Usability: The system should be easy for end-users to learn and operate effectively.
*(Note: Only two were required)*

(c) Straight vs. Phased File Conversion:

Straight (Direct) Conversion Phased Conversion
The old system is completely stopped, and the new system starts immediately. All data is converted at once before the new system goes live. The new system is implemented in stages or modules. Data conversion happens incrementally as each phase/module is rolled out.
Higher risk if the new system fails, as there's no fallback. Lower risk, as problems can be isolated to specific phases/modules. Allows gradual user adaptation.
Shorter conversion period, potentially less costly if successful. Longer overall conversion period, potentially more complex to manage the coexistence of old and new parts.

Question 4 (20 marks)

(a) Outline two advantages of using CASE tools during system analysis. (2 marks)

(b) (i) Explain the term break-even point as used in feasibility study. (2 marks)

(ii) Distinguish between evolutionary and throw away prototypes. (4 marks)

(c) With the aid of a diagram, describe the stages in system development cycle using the waterfall model. (6 marks)

(d) A system analyst conducted a feasibility study on a project and concluded the following; the system would not be operationalized in year 0, it would be implemented in year 1 and the profit for that year would be shs. 60, 000. The benefits would increase by shs. 15, 000 from year 2 to year 5. The overall development cost would be shs. 158, 000 while running costs would be shs. 8,000 per year.

Using Return on Investment (ROI), determine the viability of this project. (6 marks)

Answer

(a) Advantages of CASE Tools in System Analysis:

  1. Improved Productivity: Automate diagramming, documentation generation, and consistency checks, speeding up the analysis process.
  2. Enhanced Quality & Consistency: Enforce modeling standards, detect errors early, and maintain a central repository for consistent documentation.
  3. Better Communication: Provide standardized visual models (like DFDs, ERDs) that facilitate understanding among team members and stakeholders.
*(Note: Only two were required)*

(b) (i) Break-Even Point:

  • In an economic feasibility study, the break-even point is the point in time (or volume of activity) at which the total accumulated benefits of a project equal the total accumulated costs. After this point, the project starts generating a net profit.

(ii) Evolutionary vs. Throwaway Prototypes:

Evolutionary Prototype Throwaway Prototype
Developed iteratively and refined based on feedback, eventually becoming the final system or part of it. Built quickly to explore requirements or design ideas, then discarded. Not intended to be part of the final system.
Focuses on building a working, albeit incomplete, version of the system. Focuses on clarifying specific aspects or user interface elements.
Requires more robust initial build quality. Can be built with less focus on underlying architecture or code quality.

(c) Waterfall Model Stages:

    +-----------------------+
    |  Requirements         | Feasibility Study, Analysis
    +-----------┬-----------+
                │
                ▼
    +-----------┴-----------+
    |  System Design        | Logical & Physical Design
    +-----------┬-----------+
                │
                ▼
    +-----------┴-----------+
    |  Implementation       | Coding, Unit Testing
    +-----------┬-----------+
                │
                ▼
    +-----------┴-----------+
    |  Testing              | Integration, System, Acceptance Testing
    +-----------┬-----------+
                │
                ▼
    +-----------┴-----------+
    |  Deployment           | Installation, Conversion
    +-----------┬-----------+
                │
                ▼
    +-----------┴-----------+
    |  Maintenance          | Ongoing Support & Enhancement
    +-----------------------+

Description: The Waterfall model is a sequential development process where each phase must be fully completed before the next begins. Progress flows steadily downwards (like a waterfall) through Requirements, Design, Implementation, Testing, Deployment, and Maintenance. There is typically limited overlap or iteration between phases.

(d) Return on Investment (ROI) Analysis:

1. Calculate Total Benefits over 5 years:

  • Year 1: 60,000
  • Year 2: 60,000 + 15,000 = 75,000
  • Year 3: 75,000 + 15,000 = 90,000
  • Year 4: 90,000 + 15,000 = 105,000
  • Year 5: 105,000 + 15,000 = 120,000
  • Total Benefits = 60k + 75k + 90k + 105k + 120k = 450,000

2. Calculate Total Costs over 5 years:

  • Development Cost (Year 0/1): 158,000
  • Running Costs (Years 1-5): 8,000 * 5 = 40,000
  • Total Costs = 158,000 + 40,000 = 198,000

3. Calculate Net Profit:

  • Net Profit = Total Benefits - Total Costs
  • Net Profit = 450,000 - 198,000 = 252,000

4. Calculate ROI:

  • ROI = (Net Profit / Total Costs) * 100%
  • ROI = (252,000 / 198,000) * 100%
  • ROI ≈ 1.2727 * 100%
  • ROI ≈ 127.27%

Conclusion: With an ROI of approximately 127.27% over 5 years, the project appears financially viable based on this method.

Question 5 (20 marks)

(a) (i) Explain the term acceptance testing as used in System Development. (2 marks)

(ii) A company uses a form to capture data from its potential customers. Recently, several complaints were received from customers with regard to their data being found inaccurate. Outline four reasons for these complaints. (4 marks)

(b) UBZ Company implemented a new system in one department while the other departments continued to use the old system.

(i) Identify this system implementation method. (1 mark)

(ii) Outline three advantages of the method identified in (i). (3 marks)

(c) Melissa used observation to collect data during system investigation.

(i) Outline two challenges of using this method only. (2 marks)

(ii) Describe two other appropriate methods she could use to verify the data collected. (4 marks)

(d) Clifford prepared a Gantt chart for a proposed ICT project. Explain two significance of this tool to the project team. (4 marks)

Answer

(a) (i) Acceptance Testing:

  • Final phase of testing where end-users (or clients) test the system in a realistic environment to determine if it meets their business requirements and is acceptable for deployment.

(ii) Reasons for Inaccurate Customer Data:

  1. Data Entry Errors: Mistakes made by customers or staff when inputting data into the form (typos, incorrect information).
  2. Poor Form Design: Confusing layout, unclear instructions, or lack of validation rules on the form leading to incorrect entries.
  3. Data Processing Errors: Issues during the transfer, storage, or processing of the captured data after submission.
  4. Outdated Information: Customers' details might have changed (e.g., address, phone number) since the data was initially captured, and it wasn't updated.

(b) System Implementation Method:

(i) Identification: Pilot Implementation (or Pilot Conversion)

(ii) Advantages of Pilot Implementation:

  1. Reduced Risk: Issues or failures are contained within the pilot department, minimizing impact on the entire organization.
  2. Feedback & Refinement: Allows gathering feedback from a smaller group to identify problems and make improvements before full rollout.
  3. Training Ground: The pilot department serves as a learning experience for both users and the support team.

(c) Data Collection Verification:

(i) Challenges of Observation Only:

  1. Observer Bias: The observer's interpretation might be subjective or influenced by preconceived notions.
  2. Hawthorne Effect: People may behave differently when they know they are being observed, leading to unrepresentative data.
  3. Limited Scope: Observation might not capture underlying reasons, infrequent events, or mental processes.
*(Note: Only two were required)*

(ii) Methods to Verify Observation Data:

  1. Interviews: Talk to the people observed to confirm interpretations, understand motivations, and ask about exceptions or details not seen.
  2. Document Analysis: Review existing procedures manuals, reports, or forms related to the observed process to cross-reference findings.
  3. Questionnaires: Distribute surveys to a wider group involved in the process to confirm patterns or gather quantitative data related to observations.
*(Note: Only two were required)*

(d) Significance of Gantt Chart to Project Team:

  1. Visual Schedule Representation: Provides a clear, easy-to-understand visual timeline of project tasks, their durations, start/end dates, and dependencies.
  2. Progress Tracking: Allows the team to easily see planned vs. actual progress for each task, helping to identify delays and monitor overall project status.
  3. Communication & Coordination: Serves as a common reference point for the team to understand task assignments, deadlines, and how their work fits into the overall project plan.
*(Note: Only two were required)*

Question 6 (20 marks)

(a) Outline two measures that could be considered to deal with rapid technology change in Information Systems. (2 marks)

(b) (i) Describe two contents of a technical documentation. (4 marks)

(ii) A company intends to acquire an Information System. Describe two acquisition methods the company could use. (4 marks)

(c) Dany designed a database for an Information System. Describe two database models he could have used. (4 marks)

(d) Ufumo Dairies is a retail store that offer goods such as cereals and animal feeds to dairy farmers who are members of Wakulima Farmers’ Cooperative Society. A new member is required to provide their details which are stored in the members file. The details are verified against a file link from Cooperative Society system. Once verified a member is issued with a membership card and a notification sent to his/her phone. A member uses the card to purchase goods on credit and this information is stored in a credit file. The amount of money owed is computed monthly and sent to the Cooperative Society system. The Society deducts the money against the milk deliveries and sends a payment schedule. The member is notified to the phone and credit file is updated.

Draw a level 1 data flow diagram to represent this logic. (6 marks)

Answer

(a) Dealing with Rapid Technology Change:

  1. Modular Design: Building systems with independent components allows easier replacement or upgrading of specific parts as technology evolves, without overhauling the entire system.
  2. Adopt Open Standards: Using widely accepted standards for hardware, software, and communication protocols increases compatibility and makes it easier to integrate new technologies or switch vendors.
  3. Continuous Learning & Training: Investing in ongoing training for IT staff to keep their skills updated with new technologies.
*(Note: Only two were required)*

(b) (i) Contents of Technical Documentation:

  1. System Architecture: Diagrams and descriptions of the overall structure, components, modules, and their interactions.
  2. Database Design: ERDs, schema definitions, data dictionary, descriptions of tables, fields, and relationships.
  3. Algorithm/Logic Descriptions: Explanations of complex processes, calculations, or business rules implemented in the code.
  4. API Documentation: Specifications for how different software components or external systems interact.
*(Note: Only two were required)*

(ii) System Acquisition Methods:

  1. In-house Development: Building the system using the company's own IT staff and resources. Offers full control but can be costly and time-consuming.
  2. Purchase Off-the-Shelf Software (COTS): Buying a pre-built software package from a vendor. Faster implementation but may require customization or process changes.
  3. Leasing: Renting hardware or software for a specific period. Lower initial cost but no ownership.
  4. Outsourcing: Hiring an external company to develop or manage the system. Access to expertise but less direct control.
*(Note: Only two were required)*

(c) Database Models:

  1. Relational Model: Organizes data into tables (relations) with rows and columns. Relationships are maintained through primary and foreign keys. Standard model used by most modern databases (SQL).
  2. Object-Oriented Model: Stores data as objects, similar to object-oriented programming. Objects encapsulate data and behavior (methods). Can handle complex data types well.
  3. Hierarchical Model: Organizes data in a tree structure with parent-child relationships (one-to-many). Older model, less flexible.
  4. Network Model: Similar to hierarchical but allows more complex relationships (many-to-many) through pointers. Also an older, less common model today.
*(Note: Only two were required)*

Question 7 (20 marks)

(a) (i) Outline two challenges a user could experience when using an Information System without an accompanying manual. (4 marks)

(ii) Explain the term coupling as used in system design. (2 marks)

(b) Differentiate between condition stub and action stub as used in a decision table. (4 marks)

(c) Marvel Company made some changes to their information system to accommodate a new product.

(i) Identify the type of system maintenance used. (1 mark)

(ii) Explain two disadvantages of the maintenance identified in (i). (4 marks)

(d) A student is classified either as needy or not needy. If a student is not needy, no bursary is awarded. If needy, student discipline record and performance is considered. A needy student whose performance is a Credit and above is awarded Shs. 20,000 bursary, otherwise the student is awarded Shs. 10,000. A student with discipline issues is not awarded a bursary.

Draw a decision tree to represent this logic. (5 marks)

Answer

(a) (i) Challenges of Using IS Without a Manual:

  1. Difficulty Learning Features: Users struggle to understand how to perform tasks or use specific functionalities, leading to frustration and underutilization.
  2. Increased Errors: Without guidance, users are more likely to make mistakes, potentially leading to incorrect data or system issues.
  3. Troubleshooting Problems: Users have no reference to consult when encountering errors or unexpected behavior, increasing reliance on support staff.
*(Note: Only two were required)*

(ii) Coupling:

  • A measure of the degree of interdependence between software modules. Low coupling (modules are independent) is desirable, as it makes the system easier to understand, modify, and maintain. High coupling means changes in one module are likely to impact others.

(b) Condition Stub vs. Action Stub in Decision Table:

Condition Stub Action Stub
The upper-left quadrant of a decision table. The lower-left quadrant of a decision table.
Lists the relevant conditions or factors that influence the decision. Lists the possible actions that can be taken as a result of the conditions.
Example: "Is payment overdue?", "Is customer new?" Example: "Send reminder notice", "Offer discount", "Reject order".

(c) System Maintenance for New Product:

(i) Type of Maintenance: Adaptive Maintenance

(ii) Disadvantages of Adaptive Maintenance:

  1. Increased Complexity: Adding features can make the system more complex and harder to manage over time.
  2. Potential for New Errors: Introducing changes can inadvertently create new bugs or conflicts with existing functionality (regression).
  3. Cost and Time: Adapting the system requires resources (developer time, testing) which adds to the overall cost of ownership.
*(Note: Only two were required)*

Answer

(a) Attribute in Database Design:

  • A property or characteristic of an entity. In a relational database, attributes correspond to the columns in a table. For example, for a 'Student' entity, attributes could be `StudentID`, `Name`, `DateOfBirth`.

(b) (i) Importance of Post-Implementation Review:

  1. Assess Success: Evaluate whether the system met its original objectives and delivered the expected benefits.
  2. Identify Lessons Learned: Document what went well and what didn't during the project to improve future projects.
  3. Determine User Satisfaction: Gauge how well the system meets the needs and expectations of the end-users.
  4. Check Performance: Verify if the system operates according to performance and technical specifications.

(ii) Circumstances to Stop an Ongoing ICT Project:

  1. Major Scope Creep/Cost Overruns: If the project significantly exceeds its budget or timeline with no clear end in sight.
  2. Changing Business Needs: If the business environment or requirements change so drastically that the project is no longer relevant or viable.
  3. Technical Impracticability: If insurmountable technical challenges arise that prevent the project from being completed successfully.
  4. Lack of Continued Sponsorship/Support: If key stakeholders withdraw funding or support for the project.

(c) System Approach:

(i) Name: Use Case Analysis (Part of Object-Oriented or UML approach)

(ii) Properties of Use Case Analysis:

  1. Actor-Centric: Focuses on the interactions between external users (actors) and the system.
  2. Goal-Oriented: Each use case represents a specific goal the actor wants to achieve using the system.
  3. Behavioral Description: Describes the sequence of interactions (steps) between the actor and the system to achieve the goal.
  4. Functional Requirements Capture: Primarily used to identify and document the functional requirements of the system from a user's perspective.
