(b) Source Code vs. Object Code:

  • Source Code: This is the human-readable program code written by a programmer in a specific programming language (like Pascal, C, Java). It follows the syntax rules of that language and expresses the logic of the program.
  • Object Code: This is the machine-readable code generated by a compiler after translating the source code. It consists of low-level instructions (often machine code or an intermediate bytecode) that the computer's processor (or a virtual machine) can understand and execute. It is not easily readable by humans.

Question 2 (20 marks)

(a) Given the following C program expression:

basicpay + allowances > 22 && code == 1 || netpay > 1000

Write the order of operator evaluation. (4 marks)

(b) Differentiate between pseudocode and structure narratives as used in programming. (4 marks)

(c) Write a program in C language that accepts 10 numbers, sums them up and displays the numbers in reverse order. (5 marks)

(d) (i) Outline two functions of a compiler as used in programming. (2 marks)

(ii) Write a program in Pascal language that prompts a user to enter a price of an item ranging between 10 and 100 shillings. The maximum number of items one could buy is 20. The program then computes and displays the cost of the items. (5 marks)

Answer

(a) C Operator Evaluation Order:

Given the expression: basicpay + allowances > 22 && code == 1 || netpay > 1000

The order of evaluation based on C operator precedence and associativity (typically left-to-right for operators of the same precedence level shown here) is:

  1. + (Addition): basicpay + allowances is calculated first. Let's call the result sum1.
  2. > (Greater Than): sum1 > 22 is evaluated. Let's call the boolean result cond1.
  3. == (Equality): code == 1 is evaluated. Let's call the boolean result cond2.
  4. > (Greater Than): netpay > 1000 is evaluated. Let's call the boolean result cond3.
  5. && (Logical AND): cond1 && cond2 is evaluated. Let's call the boolean result cond4.
  6. || (Logical OR): cond4 || cond3 is evaluated. This gives the final boolean result of the entire expression.

Summary of order: +, > (left), ==, > (right), &&, ||.

(b) Pseudocode vs. Structure Narratives:

  • Pseudocode: An informal, high-level description of the operating principle of a computer program or other algorithm. It uses the structural conventions of a normal programming language (like keywords for loops, conditions, assignments) but is intended for human reading rather than machine compilation. It focuses on the logic and steps without worrying about strict language syntax.
  • Structure Narratives (Structured English): Uses a limited vocabulary of keywords (e.g., IF-THEN-ELSE, DO-WHILE, REPEAT-UNTIL, SEQUENCE) and English-like statements to describe procedural logic. It's more formal than plain English but less formal and less code-like than pseudocode. It emphasizes the control structures (sequence, selection, iteration).

(c) `do...while` vs. `while` loops (C):

  • while loop (Pre-test): The condition is checked *before* the loop body is executed. If the condition is initially false, the body is skipped entirely. The body executes zero or more times.
  • do...while loop (Post-test): The loop body is executed *first*, and then the condition is checked. This guarantees the body executes at least once, even if the condition is initially false. The body executes one or more times.

(d) C Program Interpretation:

The program demonstrates passing parameters by reference (using pointers) in C.

  1. Integer variables `i` and `j` are declared in `main`.
  2. `i` is initialized to 12, `j` is initialized to 10.
  3. The function `fn` is called with the *memory addresses* of `i` and `j` (`&i`, `&j`).
  4. Inside `fn`, the parameters `m` and `n` receive these addresses.
  5. `(*m)++` dereferences the pointer `m` (accesses the value at the address it holds - which is `i`'s value) and increments it. So, `i` in `main` becomes 13.
  6. `(*n)++` dereferences the pointer `n` (accesses `j`'s value) and increments it. So, `j` in `main` becomes 11.
  7. `fn` prints the incremented values it sees through the pointers: "13, 11".
  8. Control returns to `main`.
  9. `main` prints the current values of `i` and `j`, which have been modified by `fn`: "13 11".

(a) (ii) Errors Likely After Compilation:

Errors detected *after* successful compilation are typically:

  • Runtime Errors: Errors that occur during the actual execution of the program. They are not detected by the compiler because they often depend on specific input data or runtime conditions. Examples include division by zero, attempting to access memory that hasn't been allocated (e.g., null pointer dereference), file not found errors, or running out of memory.
  • Logic Errors: Errors where the program compiles and runs without crashing, but produces incorrect or unexpected results because the underlying algorithm or logic implemented by the programmer is flawed. The compiler cannot detect these as the syntax is correct, but the program doesn't do what was intended.

(c) Algorithm for Sequential Search:

  1. START
  2. Get the array of elements (e.g., `dataArray`) and its size (`arraySize`).
  3. Get the value to search for (`searchValue`).
  4. Initialize an index variable (`index`) to 0 (or the starting index of the array, e.g., 1 in Pascal).
  5. Initialize a flag variable (`found`) to false.
  6. WHILE `index` is less than `arraySize` AND `found` is false:
  7. IF `dataArray[index]` equals `searchValue` THEN:
  8. Set `found` to true.
  9. ELSE:
  10. Increment `index` by 1.
  11. END IF
  12. END WHILE
  13. IF `found` is true THEN:
  14. Display "Value found at index: " + `index`.
  15. ELSE:
  16. Display "Value not found in the array".
  17. END IF
  18. STOP

(Alternatively, using pseudocode keywords)

PROCEDURE SequentialSearch(dataArray, arraySize, searchValue)
  SET index = 0
  SET found = FALSE
  WHILE index < arraySize AND found = FALSE DO
    IF dataArray[index] = searchValue THEN
      SET found = TRUE
    ELSE
      INCREMENT index
    END IF
  END WHILE

  IF found = TRUE THEN
    PRINT "Value found at index ", index
  ELSE
    PRINT "Value not found"
  END IF
END PROCEDURE

Question 5 (20 marks)

(a) Outline four characteristics of quicksort technique. (4 marks)

(b) Describe two causes of errors likely to occur when a C program fails to read a data file. (4 marks)

(c) Write a Pascal program that prompts a user to enter a digit. The program then computes and displays the square root of the digit using a predefined function. (6 marks)

(d) Draw a program flowchart that could be used to solve the values of X in a quadratic equation: f(x)=ax²+bx+c. (6 marks)

Answer

(a) Characteristics of Quicksort:

  • Divide and Conquer: It's a divide-and-conquer algorithm. It works by selecting a 'pivot' element from the array and partitioning the other elements into two sub-arrays according to whether they are less than or greater than the pivot.
  • Recursive: The sub-arrays are then sorted recursively using the same quicksort algorithm.
  • In-place (typically): Common implementations sort the array in-place, meaning they require minimal additional memory space beyond the array itself (O(log n) stack space for recursion).
  • Average Time Complexity: Has an average-case time complexity of O(n log n), making it very efficient for large datasets.
  • Worst-Case Time Complexity: Has a worst-case time complexity of O(n²), which can occur if the pivot selection consistently results in highly unbalanced partitions (e.g., sorting an already sorted array with the first or last element as pivot). Randomizing pivot selection helps mitigate this.
(Any four distinct characteristics)

(b) Causes of Errors When C Program Fails to Read a Data File:

  • File Not Found: The specified file path or name is incorrect, or the file does not exist in the expected location. The `fopen()` function would typically return `NULL`.
  • Incorrect File Permissions: The program does not have the necessary operating system permissions to read the file (e.g., read access denied). `fopen()` would likely return `NULL`.
  • Incorrect File Mode: Attempting to read from a file opened in a write-only mode (e.g., using `"w"` or `"a"` instead of `"r"` or `"rb"` in `fopen()`).
  • End-of-File Reached Prematurely: Trying to read data after the end of the file has already been reached, often due to incorrect loop conditions or miscalculation of file size/record count. Functions like `feof()` can check for this.
  • File Format Mismatch: Trying to read data using a format specifier (e.g., `%d` in `fscanf`) that doesn't match the actual data type stored in the file (e.g., trying to read text as an integer).
(Any two distinct causes)

(c) Pascal Program (Square Root of Digit):

(c) Dry Run vs. Walkthrough:

  • Dry Run: A manual simulation of program execution, usually done by the programmer on paper. The programmer traces the values of variables and the flow of control step-by-step for a specific set of test inputs to verify the logic of an algorithm or code segment.

(a) Term Explanations:

  • (i) Test Data: A specific set of input values deliberately chosen to test a particular aspect or path of a program's execution. (b) (ii) Circumstances for Using Pseudocode:

  • Planning and Documentation: As part of the program design documentation to describe how specific modules or functions will work.
  • Teaching/Learning: To teach programming concepts and algorithm design without requiring knowledge of a specific programming language syntax.
(Any three distinct circumstances)

(c) Expression Evaluation:

The question seems to have two unrelated parts mixed together.

Part 1: Evaluate `j = ++i + 6` given `i=6` and `j=10`

  1. ++i (Prefix Increment): `i` is incremented *before* its value is used. So, `i` becomes 7. The value used in the expression is 7.
  2. + (Addition): The incremented value of `i` (7) is added to 6: 7 + 6 = 13.
  3. = (Assignment): The result (13) is assigned to `j`. The original value of `j` (10) is overwritten.

Result: After the statement, `i` will be 7 and `j` will be 13.

Part 2: Evaluate "post fix decrement"

This part is unclear as it's not related to the statement `j=++i + 6`. Assuming it means to evaluate `i--` (postfix decrement) *after* the previous statement (where `i` is now 7):

  • i-- (Postfix Decrement): The *current* value of `i` (7) is used first (if used in a larger expression), and *then* `i` is decremented. So, `i` becomes 6.

If it meant to evaluate `j--` (where `j` is now 13):

  • j-- (Postfix Decrement): The *current* value of `j` (13) is used first, and *then* `j` is decremented. So, `j` becomes 12.

Clarification needed on what "post fix decrement" was intended to apply to. Based on the first part, the final values are i=7, j=13.

(d) Pascal Program (Temperature Readings):

program TemperatureReadings;

const
  NUM_LOCATIONS = 4;
  NUM_TIMES = 3;

type
  LocationIndex = 1..NUM_LOCATIONS;
  TimeIndex = 1..NUM_TIMES;
  TempArray = array[LocationIndex, TimeIndex] of Real; // 2D array

var
  temperatures: TempArray;
  loc: LocationIndex;
  time: TimeIndex;

begin
  Writeln('Enter temperature readings:');

  // Read temperatures
  for loc := 1 to NUM_LOCATIONS do
  begin
    Writeln('--- Location ', loc, ' ---');
    for time := 1 to NUM_TIMES do
    begin
      Write('  Enter temperature for time ', time, ': ');
      Readln(temperatures[loc, time]);
    end;
  end;

  Writeln; // Blank line
  Writeln('--- Recorded Temperatures ---');

  // Output temperatures
  // Header
  Write('Location |');
  for time := 1 to NUM_TIMES do
  begin
    Write(' Time ', time, ' |');
  end;
  Writeln;
  Write('---------+');
  for time := 1 to NUM_TIMES do
  begin
    Write('--------+');
  end;
  Writeln;

  // Data rows
  for loc := 1 to NUM_LOCATIONS do
  begin
    Write('   ', loc:2, '    |'); // Right-align location number
    for time := 1 to NUM_TIMES do
    begin
      Write(' ', temperatures[loc, time]:6:2, ' |'); // Format temp to 2 decimal places
    end;
    Writeln;
  end;

  Readln; // Keep console window open
end.

(b) Queue Insertion (Enqueue) Flowchart:

Assumptions: Using a circular array implementation with `front`, `rear` pointers/indices, and `maxSize`. `count` tracks the number of elements.

+-------------------+
| START Enqueue(item)|
+-------------------+
    |
    V
+-------------------+      YES      +-------------------+
| Is count == maxSize?|-------------> | Display "Queue   |
| (Queue Full?)     |               | Overflow"         |
+-------------------+               +-------------------+
    | NO                                  |
    V                                     V
+-------------------+                  +-----+
| rear = (rear + 1) |                  | END |
|      % maxSize    |                  +-----+
+-------------------+
    |
    V
+-------------------+
| queue[rear] = item|
+-------------------+
    |
    V
+-------------------+
| Increment count   |
+-------------------+
    |
    V
+-----+
| END |
+-----+

Description:

  1. Check if the queue is full (number of elements `count` equals `maxSize`).
  2. If full, report an overflow error and stop.
  3. If not full, calculate the new `rear` position (wrapping around using modulo `maxSize`).
  4. Insert the new `item` at the calculated `rear` position in the queue array.
  5. Increment the element `count`.
