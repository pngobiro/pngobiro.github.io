Answer

(a) (i) Circumstances for Using Structured Programming Language:

  • Team Development: Promotes code readability and maintainability, making it easier for multiple developers to collaborate.

  • Readability Needs: When code clarity and ease of understanding are important goals.

(a) (ii) Procedural Programming:

A programming paradigm based on the concept of procedure calls. Procedures (also known as routines, subroutines, or functions) contain a series of computational steps to be carried out. The focus is on specifying the sequence of operations or commands to achieve a desired result. Programs are typically organized as a collection of these procedures.

  • (iii) Check desired results: Testing/Debugging stage.
  • (iv) Adjust to meet new demands: Maintenance stage.

(c) (i) Pseudocode for Bubble Sort:

PROCEDURE BubbleSort(dataArray, arraySize)
  // Outer loop for passes
  FOR pass = 1 TO arraySize - 1 DO
    // Inner loop for comparisons
    FOR index = 0 TO arraySize - pass - 1 DO
      // Compare adjacent elements
      IF dataArray[index] > dataArray[index + 1] THEN
        // Swap elements if out of order
        Swap(dataArray[index], dataArray[index + 1])
      END IF
    END FOR
  END FOR
END PROCEDURE

(d) (ii) Pascal Expression Evaluation:

Expression: Y = a + b * c ^ 2 - (d / e)

Issue: Standard Pascal does not have a built-in exponentiation operator (`^`). Assuming `c ^ 2` means `c * c` or `Sqr(c)`. Also, `e` is not defined in the question. Assuming `e=c=7` based on Q1(c).

Modified Expression: Y = a + b * Sqr(c) - (d / c)

Given: a=10, b=23, c=7, d=35

Order of Operations: Parentheses, `Sqr`, `*`, `/` (L-R), `+`, `-` (L-R).

  1. Parentheses: `(d / c)` -> `(35 / 7)` = 5.0
  2. Function/Square: `Sqr(c)` -> `Sqr(7)` = 49
  3. Expression becomes: `Y = 10 + 23 * 49 - 5.0`
  4. Multiplication: `23 * 49` = 1127
  5. Expression becomes: `Y = 10 + 1127 - 5.0`
  6. Addition/Subtraction (L-R):
    • `10 + 1127` = 1137
    • `1137 - 5.0` = 1132.0

Result: Y = 1132.0 (assuming `e=c` and `^2` means square)

(a) (i) Circumstances Causing Runtime Errors:

  • Division by Zero: Attempting to divide a number by zero during execution.
  • Invalid Memory Access: Trying to read from or write to a memory location that the program doesn't own (e.g., dereferencing a null or uninitialized pointer, accessing an array out of bounds).
  • File Not Found: Trying to open a file for reading that does not exist.
  • Resource Exhaustion: Running out of memory or other system resources needed by the program.
  • Invalid Input: Receiving input data that cannot be processed correctly (e.g., text when a number is expected, leading to conversion failure).
(Any two)

(a) (ii) Techniques to Prevent Unexpected Termination (Error Handling):

  • Input Validation: Checking user input or data read from files/external sources to ensure it is within expected ranges, formats, and types before processing it. Rejecting or sanitizing invalid data.
  • Checking Return Codes/Status Flags: Explicitly checking the return values of functions (especially system calls, library functions for I/O, memory allocation) that indicate success or failure. Branching logic based on the return code to handle potential errors gracefully.
  • Exception Handling (where available): Using language features like `try...catch` blocks (in C++, Java, Python, etc.) to isolate code that might fail and define specific handlers to execute when particular errors occur, preventing a program crash.
  • Defensive Programming: Writing code that anticipates potential problems, such as checking for null pointers before dereferencing them, checking array bounds before access, ensuring divisors are non-zero before division.
(Any two techniques explained)

(d) Pass by Value vs. Pass by Reference:

  • Pass by Value: A copy of the actual parameter's value is passed to the function. Changes made to the formal parameter inside the function do not affect the original actual parameter outside the function.
  • Pass by Reference: The memory address of the actual parameter is passed to the function. Changes made to the formal parameter (by dereferencing the address) inside the function *do* affect the original actual parameter outside the function.

Answer

(a) (i) Efficiency (Data Structures):

Efficiency refers to how effectively a data structure uses computational resources, primarily time (for operations like search, insert, delete) and memory space. An efficient data structure performs its operations quickly (low time complexity) and uses minimal memory (low space complexity) relative to the amount of data stored and the operations performed.

(a) (ii) Programming Approaches:

  • I. Top-Down: Starts with the overall problem and breaks it down into smaller, more manageable sub-problems. These sub-problems are further refined until they are simple enough to be coded directly. Focuses on high-level structure first.
  • II. Bottom-Up: Starts by implementing the lowest-level, basic functions or utilities. These are then combined to build larger, more complex modules, eventually forming the complete program. Focuses on building blocks first.

(d) Tree vs. Graph:

  • Tree: A hierarchical data structure consisting of nodes connected by edges, with a single root node and no cycles. Each node (except the root) has exactly one parent.
  • Graph: A more general data structure consisting of a set of nodes (vertices) and a set of edges connecting pairs of nodes. Graphs can have cycles, multiple paths between nodes, and nodes can have multiple "parents" or no specific hierarchy. Trees are a specific type of acyclic graph.

(a) (ii) Alpha vs. Beta Tests:

  • Alpha Testing: Performed *internally* by the development team or dedicated testers within the organization before the software is released externally. Its goal is to find bugs and usability issues in a controlled environment.
  • Beta Testing: Performed by a limited number of *external* users (real users in their own environments) before the final release. Its goal is to get feedback on usability, performance, and compatibility in real-world scenarios and uncover bugs missed during internal testing.

(d) Circumstances for Using an Interpreter:

  • Rapid Development/Prototyping: Interpreters allow for faster development cycles as there's no separate compilation step; code can be written and run immediately. Good for scripting and trying out ideas quickly.
  • Platform Independence: Interpreted languages (like Python, Java bytecode) can often run on different operating systems and hardware without recompilation, as long as the interpreter is available for that platform.
  • Debugging Ease: Interpreters often provide better runtime error messages and debugging environments, making it easier to find errors during execution.
  • Dynamic Typing: Many interpreted languages use dynamic typing, which can offer more flexibility during development (though potentially leading to runtime errors).
(Any two)

(c) General Syntax for Pascal `case` Statement:

CASE selectorExpression OF
  constantList1: statement1;
  constantList2: statement2;
  ...
  constantListN: statementN
[ ELSE // Optional part
  elseStatement(s); ]
END; // End of CASE statement

Where:

  • selectorExpression: An expression of an ordinal type (integer, char, boolean, enumerated, subrange).
  • constantListX: One or more constants of the same type as the selector, separated by commas, or a subrange (e.g., `1, 3, 5` or `'A'..'F'` or `10..20`).
  • statementX: The statement(s) to execute if the selector matches a constant in the corresponding list.
  • ELSE (Optional): Executes if the selector does not match any of the preceding constant lists.
  • END;: Marks the end of the CASE structure.

program CourseLevelAdvisor;

  SysUtils; // For AnsiUpperCase

(c) (i) Overflow Error (Data Structures):

An overflow error occurs when an attempt is made to add an element to a data structure that is already full and cannot accommodate more elements. This is common in fixed-size structures like static arrays used to implement stacks or queues. For example, trying to `Push` onto a full stack.

Answer

(a) (i) Record Definition:

A record (or struct in C) is a composite data structure that groups together related data items (fields or members) which can be of different data types, under a single name. It allows representing a real-world entity with multiple attributes.

(Similar to 2018j Q7b)

(a) (ii) Referencing Fields in Pascal Record:

  1. Dot Notation: Accessing a field directly using the record variable name followed by a dot (`.`) and the field name. 
    var myStudent: TStudent;
...
myStudent.name := 'Alice';
Writeln(myStudent.age);
  2. `WITH` Statement: Allows accessing fields within a specific record variable without repeatedly typing the record variable name. The fields are referenced directly within the `WITH...DO` block. 
    var myStudent: TStudent;
...
WITH myStudent DO
begin
  name := 'Bob'; // Equivalent to myStudent.name := 'Bob';
  age := 21;     // Equivalent to myStudent.age := 21;
  Writeln(Gender); // Equivalent to Writeln(myStudent.Gender);
end;

(b) Algorithm for Linear Search (Structured English):

PROCEDURE LinearSearch(dataArray, arraySize, targetValue)
  SET found = FALSE
  SET index = 0 // Start from the first element

  WHILE index < arraySize AND found = FALSE DO
    IF dataArray[index] = targetValue THEN
      SET found = TRUE
      // Optional: Store index where found
      SET foundIndex = index
    ELSE
      INCREMENT index
    END IF
  END WHILE

  IF found = TRUE THEN
    PRINT "Target value ", targetValue, " found at index ", foundIndex
  ELSE
    PRINT "Target value ", targetValue, " not found in the array"
  END IF
END PROCEDURE

(c) Stack vs. Queue (Removing Element):

  • Stack (Pop): Removes the element that was *most recently added* (Last-In, First-Out - LIFO). The element is removed from the 'top' of the stack.
  • Queue (Dequeue): Removes the element that was *added earliest* (First-In, First-Out - FIFO). The element is removed from the 'front' (or 'head') of the queue.

(d) C Program (Subtract Smaller from Larger):

#include <stdio.h>

int main() {
    int num1, num2;
    int larger, smaller, difference;

    printf("Enter the first integer: ");
    if (scanf("%d", &num1) != 1) {
        printf("Invalid input.\n");
        return 1;
    }

    printf("Enter the second integer (must not be equal to the first): ");
    if (scanf("%d", &num2) != 1) {
        printf("Invalid input.\n");
        return 1;
    }

    // Check if numbers are equal
    if (num1 == num2) {
        printf("Error: The two integers must not be equal.\n");
        return 1;
    }

    // Determine larger and smaller, calculate difference
    if (num1 > num2) {
        larger = num1;
        smaller = num2;
    } else {
        larger = num2;
        smaller = num1;
    }
    difference = larger - smaller;

    // Display results
    printf("\nThe larger integer is: %d\n", larger);
    printf("The difference is: %d\n", difference);

    return 0;
}
